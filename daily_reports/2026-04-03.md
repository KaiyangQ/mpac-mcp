### 6. AI Agent 端到端测试

接入真实 Claude API（claude-sonnet-4-20250514），创建了两个 AI agent 通过 MPAC 协议协作的 demo：

**场景**：一个 Flask Web 应用有多个已知问题。两个 AI agent（Alice：安全工程师；Bob：代码质量工程师）独立决定修什么、怎么修。

**协议流程执行情况**：

| 阶段 | 协议行为 | 结果 |
|------|---------|------|
| Session Join | HELLO → SESSION_INFO | ✅ 两个 agent 成功加入 |
| Intent Announce | 各自调用 Claude 决定工作内容 | Alice: 修 token 过期验证 (`auth.py`, `auth_middleware.py`)；Bob: 重构 auth 消除重复 (`auth.py`, `auth_middleware.py`, `models.py`) |
| Conflict Detection | Coordinator 自动 scope overlap 检测 | ✅ 检测到 `auth.py` + `auth_middleware.py` 重叠，生成 CONFLICT_REPORT |
| Conflict Negotiation | 各自调用 Claude 决定冲突响应 | Alice: "安全修复是 critical，愿意先做"；Bob: "Alice 的安全修复更紧急，我等她做完再重构" |
| Resolution | RESOLUTION 消息 | ✅ 两个 AI 独立达成一致，冲突解决 |
| Operation Commit | 各自调用 Claude 规划操作 → OP_COMMIT | ✅ Alice patch token 验证，Bob replace 合并认证逻辑 |

**关键观察**：两个 AI 的协商决策完全由 LLM 自主做出——协议只提供了结构化的冲突信息和协商通道，决策逻辑留给 agent。这验证了 MPAC "协调语义层"的核心设计理念：协议不替代 agent 的智能，只为智能协作提供结构。

总消息量：10 条 MPAC 消息（2 HELLO + 2 SESSION_INFO + 2 INTENT_ANNOUNCE + 1 CONFLICT_REPORT + 1 RESOLUTION + 2 OP_COMMIT）

## Session 2：协议覆盖度补全

在 Session 1 的基础上，将参考实现从 ~30% 覆盖度提升至 ~85%，实现了三大功能组：

### 7. Intent Expiry Cascade + Conflict Auto-Dismiss

**Intent Expiry Cascade（Section 15.7）**：
- Intent 到达 TTL 后自动过期，状态机转换 ACTIVE → EXPIRED
- 级联影响：PROPOSED/FROZEN 操作自动 REJECT，COMMITTED 操作不受影响
- 实现了 `check_intent_expiry()` 方法，支持批量过期检测

**Conflict Auto-Dismiss（Section 17.9）**：
- 当冲突关联的所有 intent 和 operation 都到达终态时，冲突自动 DISMISS
- 排除条件：如果存在 COMMITTED 操作，不自动 dismiss（需要人工确认）
- 实现了 `check_auto_dismiss()` 方法

Python 测试：11 个用例（6 expiry + 5 dismiss）；TypeScript 测试：10 个用例（5 + 5）

### 8. HEARTBEAT + GOODBYE + Liveness 检测

**Heartbeat 机制**：
- 参与者定期发送 HEARTBEAT（默认间隔 30s），包含 status（idle/working/blocked/awaiting_review/offline）、active_intent_id、summary
- Coordinator 跟踪每个参与者的 last_seen 时间戳和 is_available 状态

**Liveness 检测**：
- `check_liveness()` 扫描所有参与者，超过 unavailability_timeout_sec（默认 90s）未心跳则标记为不可用
- 不可用触发级联：广播 PROTOCOL_ERROR（PARTICIPANT_UNAVAILABLE）→ 悬挂其 intent（ACTIVE → SUSPENDED）→ 冻结其 PROPOSED 操作（→ FROZEN）→ 放弃其 FROZEN 操作（→ ABANDONED）
- offline status 的参与者豁免 liveness 检测

**GOODBYE 协议**：
- 参与者主动离开时发送 GOODBYE，携带 reason 和 intent_disposition（withdraw/transfer/expire）
- withdraw：立即撤回所有活跃 intent
- expire：保留 intent 的 TTL，到期后自然过期

**断线恢复**：
- 参与者重新发送 HELLO 或 HEARTBEAT 时，自动恢复其 SUSPENDED intent（→ ACTIVE）

Python 测试：9 个用例；TypeScript 测试：7 个用例

### 9. CONFLICT_ACK + CONFLICT_ESCALATE + Arbiter 工作流

**冲突确认（CONFLICT_ACK）**：
- 参与者确认已收到冲突通知，冲突状态 OPEN → ACKED
- ack_type 支持 acknowledged/reviewing/will_resolve

**冲突升级（CONFLICT_ESCALATE）**：
- 参与者可将冲突升级到仲裁者，冲突状态 OPEN/ACKED → ESCALATED
- 记录 escalated_to、escalated_at、reason、context

**Arbiter 仲裁**：
- 仲裁者（role=arbiter）通过 RESOLUTION 消息解决冲突
- 支持 approve/reject/merge/defer/dismiss 五种决策

**Resolution Timeout + Auto-escalation**：
- `check_resolution_timeouts()` 检查所有 OPEN/ACKED 冲突是否超过 resolution_timeout_sec（默认 300s）
- 超时自动升级到 arbiter；无 arbiter 则广播 PROTOCOL_ERROR（RESOLUTION_TIMEOUT）

Python 测试：7 个用例；TypeScript 测试：6 个用例

### 10. INTENT_UPDATE + INTENT_CLAIM

**INTENT_UPDATE**：
- 支持更新 intent 的 objective、scope、ttl_sec
- scope 变更时自动重新检测 overlap，与现有冲突去重
- 仅 intent owner 可更新

**INTENT_CLAIM（Section 14.5.4）**：
- 当参与者不可用且其 intent 被 SUSPENDED 时，其他参与者可 claim 该 intent
- First-claim-wins 语义：第一个 claim 成功，后续 claim 返回 CLAIM_CONFLICT 错误
- 流程：验证 intent 存在 → 检查未被 claim → 验证状态为 SUSPENDED → 注册 claim → 原 intent WITHDRAWN → 创建新 ACTIVE intent → 级联处理

**断线恢复优先**：
- 如果原参与者在 claim 前重新上线，其 SUSPENDED intent 自动恢复为 ACTIVE，claim 窗口关闭

Python 测试：8 个用例；TypeScript 测试：8 个用例

### 11. 状态机扩展

**IntentStateMachine** 新增：
- SUSPENDED → ACTIVE（resume）、SUSPENDED → WITHDRAWN（withdraw）、SUSPENDED → EXPIRED（expire）

**OperationStateMachine** 新增：
- FROZEN → PROPOSED（unfreeze）、FROZEN → REJECTED（reject）、FROZEN → ABANDONED（abandon）
- FROZEN 从终态列表中移除
