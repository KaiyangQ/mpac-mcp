# 事件一：Windows bootstrap 4 bug

早上接手时 Mac 端 Claude 的分析列了 **3 个** bug。本机（KAIYANG 的 Windows / Python 3.14 / PowerShell 5.1）retest 之后发现**第 4 个**。

## Bug 1: `$ErrorActionPreference = "Stop"` + 原生命令 stderr

PowerShell 把原生命令的 stderr 行包成 `ErrorRecord`，`Stop` 模式下就算 exit 0 也杀脚本。

现场：`pip install mpac-mcp` 拉进 `pywin32`，后者装完往 stderr 写 `WARNING: scripts ... not on PATH`。当时码里的 `2>$null` 重定向在 Python 3.14 + PowerShell 5.1 这条 branch 下**反而激活** `NativeCommandError`，触发 EAP=Stop 的 `throw`，脚本当场死。

修：line 655 `Stop`→`Continue`，`2>$null` 从所有 pip-install 行删（保留 3 个 detection query 里的 `2>$null`，它们在 try/catch 里，静音是刻意的）。改完 stderr warning 直接进 console —— 对 debug 反而有用。

## Bug 2: user-site Scripts 路径算错

原码 `python -m site --user-base` + `Join-Path "Scripts"` → `%APPDATA%\Python\Scripts`（**不存在**）。

真实路径带版本后缀：`%APPDATA%\Python\Python314\Scripts`。

修：换 sysconfig 问 Python 自己算：

```powershell
$userScripts = & $python -c "import sysconfig, sys; print(sysconfig.get_path('scripts', 'nt_user' if sys.platform == 'win32' else 'posix_user'))"
```

本机实测（Python 3.14）：

```
old  C:\Users\Kaiyang\AppData\Roaming\Python\Scripts          ← 不存在
new  C:\Users\Kaiyang\AppData\Roaming\Python\Python314\Scripts  ✓
```

## Bug 3: `Test-Path $claudeDir` 误判已登录

`%USERPROFILE%\.claude` 存在不等于登录过。Claude Code CLI 自己因为别的原因（project 索引、sessions 元数据）也会创建这目录。

修：两条件 AND —— `claude` 必须在 PATH 且 `~/.claude/sessions/` 必须非空（只有真 OAuth 过才写 sessions）：

```powershell
$claudeInstalled = [bool](Get-Command claude -ErrorAction SilentlyContinue)
$sessions = Get-ChildItem -Path (Join-Path $claudeDir "sessions") -ErrorAction SilentlyContinue
$loggedIn = $claudeInstalled -and $sessions -and $sessions.Count -gt 0
```

False negative（多 prompt 一次 login）代价很低；false positive（跳过 login 让 relay 静默失败）代价很高。所以保守。

## Bug 4（retest 时冒出来）: plain pip install 也可能走 user-site

本机试 install 路径时撞到 —— install 明明退 0，但 `Get-Command mpac-mcp-relay` 找不到 → Die "not on PATH after install"。

根因：Windows + 非 admin + 全局 Python 时，**plain `pip install`（没 `--user`）pip 自己静默 fallback 到 user-site 装**，留下 stderr warning "scripts ... not on PATH" 但 exit 0。老码里 PATH-prepend 逻辑**只**在 `--user` 显式 fallback 那条分支跑，plain install 成功 → 跳过 prepend → binary 找不到。

Mac 端 Claude 的 3 bug 列表里没这条，是本地 retest 真跑才浮出来的。

修：PATH prepend 拉到 install block **外**，无条件跑（`Test-Path $userScripts` 防错）。Idempotent，其他分支里 no-op。

# 事件二：Chat 还是不回 —— mpac-mcp 0.2.2 relay 的第二层 bug

第一轮 deploy 之后 KAIYANG 浏览器输入 "hello"，弹出红色：

```
[relay] Claude Code failed (exit 1): Warning: no stdin data received in 3s, proceeding without it.
Error: Input must be provided either through stdin or as a prompt argument when using --print
```

bootstrap 修完 relay 能连（verified），但 **chat passthrough 挂在 relay → claude 子进程那一层**。这是 mpac-mcp 0.2.2 的 bug，**独立于 bootstrap**。

## 根因

`mpac-mcp/src/mpac_mcp/relay.py:188` 的 `handle_chat()` 用 `asyncio.create_subprocess_exec(claude_binary, "-p", ..., message)` —— `message` 作为**最后一个 positional argv**。

Windows 上 `claude_binary` 是 npm 装的 `claude.cmd` shim。Python subprocess 跑 `.cmd` 走 `cmd.exe /c claude.cmd ...`。**cmd.exe 的 argv quoting 是地球上最差的之一** —— `&`、`|`、`%`、`(`、`)`、embedded newline 全是特殊字符，用户 chat 输入随便打几个就会被 cmd.exe 吞。claude.exe 收到 `-p` 但没 positional，等 stdin 3s，报这个 error 回来。

Mac 上不挂因为 `claude` 在 Mac 是 `#!/usr/bin/env node` POSIX 脚本，subprocess 直接 exec，**没 cmd.exe 中间层**。

## 修：argv → stdin

```python
# BEFORE:
proc = await asyncio.create_subprocess_exec(
    ctx.claude_binary, "-p",
    ...,
    "--append-system-prompt", _SYSTEM_PROMPT,
    message,                              # ← cmd.exe 吞
    stdout=PIPE, stderr=PIPE, env=env,
)
stdout, stderr = await proc.communicate()

# AFTER:
proc = await asyncio.create_subprocess_exec(
    ctx.claude_binary, "-p",
    ...,
    "--append-system-prompt", _SYSTEM_PROMPT,
                                          # ← 删掉 message positional
    stdin=asyncio.subprocess.PIPE,        # ← 新增 stdin pipe
    stdout=PIPE, stderr=PIPE, env=env,
)
stdout, stderr = await proc.communicate(input=message.encode("utf-8"))
```

stdin 是真 OS pipe，cmd.exe 碰不到。跨平台干净（Mac 行为不变 —— Mac 从来就没走过 cmd.exe，stdin pipe 和 argv 效果一样）。

# 事件三：Bootstrap UX 收拾（一条 commit 顺手做）

事件二的 fresh-user 验证 run 输出里有两行吓人但功能无影响的 ERROR / WARNING：

```
ERROR: Could not find a version that satisfies the requirement mpac-mcp>=0.2.3 (from versions: 0.1.0 ... 0.2.2)
...
WARNING: The scripts ... are installed in '...\Python314\Scripts' which is not on PATH.
```

第一条是 PyPI CDN 刚 publish 后的 ~30s 索引 race（pip 第一次看 `/simple/mpac-mcp/` 页面还没 0.2.3；几秒后 `--user` 重试已经有了）。第二条正确但**冗余** —— 我们 PATH prepend 紧跟后面就把它加上了。对新用户又吓人又误导。

## 修：install ladder 换顺序

以前 `plain → --user → --break-system-packages`，现在 `--user → plain → --break-system-packages`：

- `--user` 对**非 venv** 的全部环境都能跑（Windows 非 admin 全局 Python、Mac brew、Linux 全局）；加 `--no-warn-script-location` 抑制那行冗余 warn
- 遇到 venv 时 `--user` 报 "Can not perform a '--user' install" → 跳 plain（写到 venv site-packages）
- 遇到 PEP 668 distro（Ubuntu 23+ 系统 Python）时两者都失败 → 第三档 `--break-system-packages`

3 档覆盖面不变，最常见 path（Windows non-admin）第一步就静默成功。

commit `ecfd2a1 bootstrap.ps1: reorder pip ladder to --user first + suppress not-on-PATH warning`。

# 事件四：OneDrive 跟所有 build 工具开战

上面三件事撞到的环境坑（纯 Windows + OneDrive Files-On-Demand 特有）：

## Turbopack vs OneDrive

Next.js 16 dev 默认 Turbopack，往 `.next/` 写 source-map → OneDrive cloud-provider 层 reject：

```
TurbopackInternalError: failed to write to "...\.next\dev\server\chunks\ssr\...map"
Caused by: The cloud operation was unsuccessful. (os error 389)
```

试过 junction `.next` → OneDrive 外，但 Turbopack 的 PostCSS resolver 找不到 node_modules。最后用 `next dev --webpack` —— webpack 的写文件语义不触发 OneDrive cloud-op。

## git append vs OneDrive

`git commit` 往 `.git/logs/refs/heads/main` append：

```
fatal: cannot update the ref 'refs/heads/main': unable to append to '.git/logs/refs/heads/main': Invalid argument
```

设 `git config --local core.logAllRefUpdates false` + `windows.appendAtomically false` + 临时移走老 reflog 才过 commit。后续 push / stash 还会偶发 mmap fail。

最终 workaround：所有 sync-deploy 从 OneDrive 外的干净 clone (`C:/temp/mpac-deploy-clone`) 跑。`.git` 不在 OneDrive 就不撞。

## pip build vs OneDrive

`pip install ./mpac-mcp`（sdist build）写 egg-info：

```
error: [Errno 22] Invalid argument
(writing top-level names to src\mpac_mcp.egg-info\top_level.txt)
```

绕法：copy mpac-mcp 源码到 `C:/temp/mpac-mcp-build/` 后 `python -m build`。

## 根源

Windows + OneDrive Files-On-Demand 对 `mmap` / atomic-rename / append 某些组合 reject。git、Turbopack、pip setuptools 全重度依赖这些 syscall → 全中。

**长远解**：repo clone 到 OneDrive 外（`C:\dev\` 之类）。今天时间紧先 workaround。

## 产品感：跨平台这件事必须上 CI

今天 5 个 bug **全是 Windows 特有**，根因可以归纳成一句话：**Windows 的工具链是 Unix-first 工具被移植时，Windows 加一层 `.cmd`/`.ps1`/OneDrive/PowerShell 包装层去模拟 Unix 行为**。模拟 90% 像，剩下 10% 在边界上漏，而**开发者只在 Mac 上测** → 这 10% 没人能复现。

MPAC 里两个组件都犯过这错：

- `bootstrap.ps1` 的 `_render_bootstrap_ps1()` 完全是 bash 版本的"手工翻译"，从没在 Windows 上跑过
- `mpac-mcp 0.2.2` 的 `handle_chat()` 用 asyncio 启 claude，没考虑 Windows 上 claude 其实是 `.cmd` shim

如果没 KAIYANG 自己切到 Windows 测，这两个 bug 到内测者身上才会炸，而内测者多半会直接放弃不来反馈，我们永远不知道。

**明天起 TODO**：`mpac-mcp` 仓库加一个 GitHub Actions，`windows-latest` runner 上跑 bootstrap + chat 端到端 smoke。12 小时能做的事，比今天修这 5 个 bug 短得多。

同样的道理也适用于 `web-app/api` 的 `_render_bootstrap_ps1()` —— 可以加一个 pytest 跑 PowerShell parser 做语法检查 + 一个 optional CI 步骤真实跑 Windows runner。
