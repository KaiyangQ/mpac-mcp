# Daily Report — 2026-04-05

## 概述

今天基于一轮独立的协议审查（protocol audit），完成了 MPAC v0.1.12 的 conformance closure。审查发现协议设计层面已经成熟，但"发布工件层面"（JSON Schema、示例、条件约束）还没有完全闭合。本轮修订专注于把 spec prose 里的 MUST / Conditional 要求编码进 JSON Schema，使第三方实现可以通过 schema 校验直接获得 wire compatibility 保障。

---

## 完成事项

### 1. 独立协议审查

对 MPAC v0.1.11 做了一轮覆盖"spec → schema → ref-impl → demo"四层的一致性审查，识别出以下问题：

- **P1** — JSON Schema 仅覆盖 13/21 message types，缺少 heartbeat、goodbye、intent_claim、intent_update、intent_withdraw、op_reject、conflict_ack、conflict_escalate
- **P1** — envelope.schema.json 的 `payload` 是泛型 `object`，没有按 `message_type` 绑定到具体 payload schema
- **P1** — COORDINATOR_STATUS 有 spec/schema/impl 漂移：schema 缺 `next_coordinator_epoch`，`event` 是自由字符串，实现发 `authorization` 事件但 spec 未定义
- **P2** — OP_BATCH_COMMIT entry schema 比 spec 松（缺少 required `state_ref_before`/`state_ref_after`）
- **P2** — INTENT_CLAIM_STATUS 和 RESOLUTION.outcome 的条件约束只在 prose 里，没有进入 schema
- **P2** — demo transcript 版本仍为 0.1.10

### 2. v0.1.12 — Conformance Closure

基于审查结果，完成了以下修复：

**新增 8 个 message payload schema：**
- `heartbeat.schema.json`：status (enum: idle/working/blocked/awaiting_review/offline)
- `goodbye.schema.json`：reason (enum: user_exit/session_complete/error/timeout)，intent_disposition
- `intent_claim.schema.json`：claim_id, original_intent_id, original_principal_id, new_intent_id, objective, scope
- `intent_update.schema.json`：intent_id + optional updates (objective, scope, assumptions, ttl_sec)
- `intent_withdraw.schema.json`：intent_id, reason
- `op_reject.schema.json`：op_id, reason
- `conflict_ack.schema.json`：conflict_id, ack_type (enum: seen/accepted/disputed)
- `conflict_escalate.schema.json`：conflict_id, escalate_to, reason

**Envelope oneOf dispatcher：**
- envelope.schema.json 新增 `allOf` 块，包含 21 个 `if/then` 子句，按 `message_type` 将 `payload` 分发到对应的 message schema
- 新增 coordinator-authored message 的 `coordinator_epoch` 条件约束

**COORDINATOR_STATUS 对齐：**
- `event` 从自由字符串改为 enum：`heartbeat`, `recovered`, `handover`, `assumed`, `authorization`
- `authorization` 事件正式标准化（之前两个参考实现已在发送，但 spec/schema 未覆盖）
- 新增 `next_coordinator_epoch` 字段 + handover `if/then` 约束
- 新增 `authorized_op_id`、`authorized_by`、`authorized_batch_id` 字段 + authorization `if/then` 约束
- 移除了 schema 中多余的 `snapshot` 字段（recovery 有独立机制）
- SPEC.md 同步更新

**OP_BATCH_COMMIT 收紧：**
- entry `required` 从 `["op_id", "target", "op_kind"]` 改为 `["op_id", "target", "op_kind", "state_ref_before", "state_ref_after"]`

**条件约束编码：**
- `INTENT_CLAIM_STATUS`：三组 `if/then` — approved→new_intent_id, rejected→reason, withdrawn→reason
- `outcome.schema.json`：rejected 非空时 requires rollback

**版本修正：**
- Demo transcript 从 0.1.10 更新到 0.1.12
- 全线版本号更新：SPEC.md、envelope.schema.json、Python (coordinator/envelope/__init__/pyproject.toml/tests)、TypeScript (coordinator/envelope/package.json/package-lock.json)

### 4. 二轮审查与修正

初版 v0.1.12 出来后做了一轮自审查，发现 3 个 conformance 问题并在同一版本内修正：

1. **Envelope `coordinator_epoch` 约束范围错误**：初版把 `CONFLICT_REPORT` 和 `RESOLUTION` 列入必须带 `coordinator_epoch` 的消息类型，但它们可以由任意 participant / resolver 发送。改为移除这两个，补入 `SESSION_CLOSE`（coordinator-only）。

2. **Authorization 事件缺 required 字段**：`COORDINATOR_STATUS` schema 要求 `coordinator_id` 和 `session_health` 必填，但两个实现的 `authorizeOperation` 方法没有填这两个字段。修正：在 Python 和 TypeScript 的 `_authorize_operation` / `authorizeOperation` 中补入 `coordinator_id`（取 `self.coordinator_id`）和 `session_health`（根据 open conflicts 计算）。

3. **Outcome rollback `if/then` 过严**：初版在 `outcome.schema.json` 加了"rejected 非空即要求 rollback"的约束，但 spec 只在 rejected 包含 COMMITTED operation 时才要求 rollback。拒绝 intent 或 pre-commit proposal 不需要回滚。移除 `if/then`，改为 description 注释 + runtime enforcement。

### 5. 三轮审查与修正

二轮修完后再次审查，发现 `OP_REJECT` 和 `PROTOCOL_ERROR` 也不是 coordinator-only（participant 可以发 `CAUSAL_GAP` 类型的 `PROTOCOL_ERROR`，owner/arbiter 可以发 `OP_REJECT`），从 `coordinator_epoch` required 列表中移除。最终列表收窄为 4 个纯 coordinator-authored 消息类型：`SESSION_INFO`、`SESSION_CLOSE`、`COORDINATOR_STATUS`、`INTENT_CLAIM_STATUS`。

同时补了两处文档同步：

1. **SPEC.md Section 14.6 `authorization` 语义段**：正文 event 枚举原来只列 4 个值，但 payload 表和 schema 已经有 5 个。补入 `authorization` 事件的完整语义描述（pre_commit 模式、所需字段、与 OP_BATCH_COMMIT 的关系），消除规范内部自相矛盾。

2. **MPAC_Developer_Reference.md 同步到 v0.1.12**：版本号从 v0.1.11 → v0.1.12，COORDINATOR_STATUS 字段表补入 `authorization` / `authorized_op_id` / `authorized_batch_id` / `authorized_by`，事件补充段加了 `authorization` 说明，交叉引用表补了两条新关联，`handover` 的条件字段从 O 改为 C。

### 6. 六项运行时执行差距修复（v0.1.12 enforcement hardening）

在前五轮修正覆盖了 schema、spec、文档层面后，进一步审查发现参考实现中有 6 项 spec normative 要求没有在运行时执行：

1. **HELLO-first gate**：未注册的 principal（除 HELLO 外）发送任何消息应返回 `AUTHORIZATION_FAILED`。GOODBYE 不再豁免——未注册 sender 无权触发 intent disposition。修复：在 `process_message` 入口添加 sender 注册检查，仅 HELLO 豁免。
2. **Credential validation**：非 `open` security profile 的 HELLO 消息必须携带有效 credential，否则返回 `CREDENTIAL_REJECTED`。修复：在 `_handle_hello` / `handleHello` 顶部添加凭证校验。
3. **Resolution authority**：pre-escalation conflict 只有 owner/arbiter 可以 resolve；post-escalation 只有 escalate_to target 或 arbiter 可以 resolve。修复：从 `_is_authorized_resolver` 中移除 `related_principal` 逻辑。
4. **Frozen-scope enforcement**：根据 Section 18.6.2，scope 仅在 `resolution_timeout_sec` 超时后进入冻结状态（而非 conflict 创建时立即冻结）。冻结后阻止 `INTENT_ANNOUNCE`、`OP_PROPOSE`、`OP_COMMIT`、`OP_BATCH_COMMIT`。修复：在 Conflict 数据结构中添加 `scope_frozen` 标志，由 `check_resolution_timeouts` 在无 arbiter 时设置；`_is_scope_frozen` helper 仅匹配已冻结的 conflict；四处检查点（含 batch）。
5. **Batch atomicity rollback**：`all_or_nothing` 批次验证失败时，已注册的 operation 必须从 coordinator state 中清除。修复：在返回 batch reject 前执行 `operations.pop(op.op_id)` 清理。
6. **Error codes 补全**：`CAUSAL_GAP` 和 `INTENT_BACKOFF` 加入 Python/TypeScript ErrorCode enum；`authorization` 加入 CoordinatorEvent enum。

**二轮审查修正（同日）：**
发现三处 P1 问题并立即修复：
1. GOODBYE 从 HELLO-first gate 豁免列表中移除（防止未注册 sender 通过 GOODBYE(active_intents) 强制 withdraw 他人 intent）
2. Frozen-scope 从"conflict 创建时立即冻结"改为"resolution_timeout 超时且无 arbiter 时才冻结"（符合 Section 18.6.1-18.6.2）
3. OP_BATCH_COMMIT 补上缺失的 frozen-scope 检查

**三轮审查修正（同日）：**
发现 2 处 P1 + 2 处 P2 问题并修复：
1. **[P1] OP_BATCH_COMMIT per-entry intent_id bypass**：frozen-scope 检查仅看顶层 `intent_id`，per-entry `intent_id` 绕过。修复：遍历所有 entry 的 `intent_id` 并逐一检查冻结状态。
2. **[P1] scope_frozen 未写入 snapshot**：coordinator recovery 后丢失冻结状态。修复：snapshot 序列化和 recovery 反序列化都包含 `scope_frozen`。
3. **[P2] INTENT_ANNOUNCE partial overlap 过严**：spec 要求"fully contained → reject, partial overlap → accept with warning"，实现对任意 overlap 都直接拒绝。修复：新增 `scope_contains` / `scopeContains` 函数 + `checkFrozenScopeForIntent` 区分完全包含 vs 部分重叠；部分重叠注册 intent 但附带 SCOPE_FROZEN warning。
4. **[P2] HELLO-first gate error code 不一致**：spec 写 `INVALID_REFERENCE`，实现返回 `AUTHORIZATION_FAILED`，测试固化了错误行为。修复：两边实现改为 `INVALID_REFERENCE`，测试同步更新。

**四轮审查修正（同日）：**
发现 frozen-scope 的根本性设计缺陷——所有检查基于 intent_id 查询，但 `intent_id` 在 schema 中是 optional：
1. **[P1] OP_BATCH_COMMIT 无 intent_id 绕过**：当 batch entry 和 envelope 均不带 `intent_id` 时，intent-based 冻结检查全部跳过。修复：改为 target-based 检查——遍历每个 entry 的 `target`，构造 file_set scope 并调用 `_is_scope_frozen`。
2. **[P1] Python OP_PROPOSE/OP_COMMIT 无 intent_id 绕过**：Python 的 frozen-scope guard 仅在 `intent_id` 存在时执行（TypeScript 已经是 target-based）。修复：Python 的 `_handle_op_propose` 和 `_handle_op_commit`（post-commit path）改为 target-based，与 TypeScript 对齐。
