### 9. 真实跨网络联调与发现的 UX 问题

用 ngrok 把本地 coordinator 暴露为公网地址（`wss://rutilated-bibulously-coraline.ngrok-free.dev`），与另一位合作者（使用 Windows + PowerShell + uv 管理的 Python 3.11）进行了真实跨网络测试。

**修复的两个 starter kit bug：**

| 问题 | 症状 | 修复 |
|------|------|------|
| `run.py` 不支持命令行参数 | 合作者习惯 `python run.py wss://...` 传地址，被忽略 | 新增 `sys.argv[1]` 处理，CLI 参数优先于 config |
| config 缺字段时崩溃 | 旧版 config.json 缺 `server` 字段，`cfg["server"]` KeyError | 改为逐字段检查并按需补问，不崩溃 |

**发现的 UX 问题：intent 生命周期与 task 生命周期不匹配**

测试场景：两人同时下达改 `auth.py` 的任务，触发 `CONFLICT_REPORT` → 协商 → 提交成功。当合作者不再操作、只有 Alice 重新输入同一个 task 时，**依然检测到冲突**。

**根因分析：**
- MPAC 协议中 `INTENT_ANNOUNCE` 的语义是"我计划在这个范围工作"，不是"我执行一次操作"
- `OP_COMMIT` 不会自动结束父 intent——协议允许一个 intent 下有多个 operation
- 所以第一轮测试后，合作者的 intent 虽然没再下任务，但在 coordinator 眼里依然是 `ACTIVE` 状态占着 `auth.py`
- Alice 重新宣告新 intent 时，与合作者的"幽灵 intent"重叠，触发冲突

**协议层面这是正确行为**，但在交互式 CLI 的场景下，用户直觉上"任务结束了 intent 就该结束"，暴露了包的 UX gap。

**修复（agent 层，不改协议）：**
- `_run_task_interactive` 在任务完成后自动发送 `INTENT_WITHDRAW`（reason=`task_completed`）
- 新 task 开始时清空 `conflicts_received` 列表，避免上一轮的 stale 冲突记录污染
- 这把"一个 task == 一个 intent"的语义绑定放在 agent 层，而不是协议层

**为什么不在协议层修复：**
- MPAC 的 intent/operation 分离是故意的——不同场景下"intent 何时结束"有不同语义（交互式 CLI vs 长周期工作流 vs 批处理）
- 协议让 coordinator 保持 "dumb"，由 agent 业务逻辑决定 intent 边界
- 这是 UX 决策，不是协议 bug

**相关未决问题（待进一步讨论）：**
- 是否应该在协议层面支持"所有 operation 完成后自动结束 intent"？涉及新增 intent 状态（COMPLETED?）、"最后一个 op" 如何判定、与 TTL/WITHDRAW 机制的关系等。属于 v0.2.0 级别的协议演进讨论。
