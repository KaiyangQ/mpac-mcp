## (2c) announce 时 auto-supersede 同 principal 重叠 intent

[mpac-package/src/mpac_protocol/core/coordinator.py:859](mpac-package/src/mpac_protocol/core/coordinator.py:859) — `_handle_intent_announce` 在注册新 intent 之前,把同 principal 重叠的旧 ACTIVE intent 转成 WITHDRAWN,顺便调 `_cascade_intent_termination` + `_check_auto_dismiss`,跟正常 withdraw 路径一致:

```python
for other in list(self.intents.values()):
    if other.intent_id == intent.intent_id: continue
    if other.principal_id != intent.principal_id: continue
    if other.state_machine.current_state != IntentState.ACTIVE: continue
    if not scope_overlap(other.scope, intent.scope): continue
    other.state_machine.transition("WITHDRAWN")
    superseded_ids.append(other.intent_id)
for sid in superseded_ids:
    responses.extend(self._cascade_intent_termination(sid))

self.intents[intent.intent_id] = intent
responses.extend(self._detect_scope_overlaps(intent))
if superseded_ids:
    responses.extend(self._check_auto_dismiss())
```

# 续事件 C — mpac-mcp 0.2.8:`--resume` 跨 turn 对话记忆

0.2.7 解决了 intent_id 这一个具体字段,但**根因没解决**:每个
`claude -p` 是独立子进程,**对话上下文整体丢失**(不只 intent_id,所
有 prior context)。用户问:"能不能让 Claude 真的记得上一轮在做什么?"

我先调研 Claude Code 自身能力(用 `claude-code-guide` agent 查文档):

* `claude --resume <session_id>` 在 `-p` 模式下**有效**
* `claude -p --output-format json` 让 stdout 输出结构化 JSON,里面带
  `session_id` 字段
* session 状态文件落在 `~/.claude/projects/<cwd>/<session_id>.jsonl`
  (用户本地,不进我们 server),30 天默认 cleanup
* 同一 session 可以续命到多个 `claude -p` invocation —— 完美

实现非常干净(~30 行核心代码):

```python
_chat_lock: Optional[asyncio.Lock] = None  # serialize concurrent chats
_session_id: Optional[str] = None          # the captured UUID

async def handle_chat(ctx, message) -> str:
    async with _get_chat_lock():
        return await _handle_chat_locked(ctx, message)

async def _handle_chat_locked(ctx, message):
    argv = [ctx.claude_binary, "-p", "--output-format", "json", ...]
    if _session_id is not None:
        argv.extend(["--resume", _session_id])
    proc = await asyncio.create_subprocess_exec(*argv, ...)
    stdout, stderr = await proc.communicate(input=message.encode())
    if proc.returncode == 0:
        data = json.loads(stdout)
        new_session_id = data.get("session_id")
        if new_session_id:
            _session_id = new_session_id    # capture for next turn
        return data.get("result", "").rstrip()
```

并发处理:WS loop 用 `asyncio.create_task` 派发 chat,可能并发 → session_id
读写竞态。加 `asyncio.Lock` 把 chat 串行化。代价:同一 relay 一时刻
只跑一个 `claude -p`(每条消息已经 5-30s,多排个队可接受);收益:
session_id 单读单写,无竞争。

**架构上的意义**:relay 不再是"无状态 RPC",变成"有跨 turn 记忆的
代理"。之前 user 跟 Claude 多轮对话每次都得重新喂背景,现在 Claude
真的记得「我们刚才在讨论 X,然后我开始做 Y,你说先停一下」这种
context。

5 个新单测覆盖:JSON 解析、`--resume` 在第二次出现、session_id 旋转
(stale resume 落空)、JSON 解析失败 fallback、`--output-format json`
进 argv。**245 测试 pass**。

1. **测试者本机 mpac-mcp 是 0.2.9 但 relay 进程是 0.2.8 启动的** ——
