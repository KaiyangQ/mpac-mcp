## 协议层先排雷

`web-app/api/mpac_bridge.py` 已有的 `mpac_smoke.py` 跑的就是 "两个 user 同文件 → 双方收 CONFLICT_REPORT"。我先把它过一遍：

读 `page.tsx:1046-1063` 的注释：

> Intent announcement is owned by Claude's MCP `announce_intent` tool, NOT by the browser. ... The browser is a viewer + chat surface, not an editor.

也就是说浏览器**不会**自己 announce intent，所有 intent 都从 mpac-mcp 那边走 `POST /api/agent/intents`。看 `routes/agent.py:248`：

```python

```

跟 `routes/main.py:206` 的 `principal_id = f"user:{user.id}"`（浏览器 WS 注册的）**不同**。

回到 `mpac_bridge.py:495-500`（修复前）：

```python
elif resp_type == "CONFLICT_REPORT":
    payload = resp.get("payload", {})
```

`CONFLICT_REPORT` **只单播给 principal_a/principal_b**。当两边都是 `agent:user-N` 时，`user:N` 的浏览器 socket 完全不在收件人里。

但 `INTENT_ANNOUNCE` 是 broadcast 的（`_BROADCAST_INBOUND_TYPES`），所以 Who's Working 看得到对方 agent 在编辑 — 这就是用户观察到的不对称。

## 用 3-socket 脚本铁证

写了 `agent_conflict_repro.py`：开 3 个 WS（Alice 浏览器 / Alice agent / Bob agent），让两个 agent 通过 `POST /api/agent/intents` 在同一文件 announce intent，drain 每个 socket，列收到的 envelope。

修复前：

```
[alice browser  (user:1)]  (5 envelopes)
    ← SESSION_INFO
    ← PARTICIPANT_UPDATE × 2
    ← INTENT_ANNOUNCE × 2          ← Who's Working 的来源
                                   ← 没有 CONFLICT_REPORT

[alice agent    (agent:user-1)]  ← CONFLICT_REPORT ✓
[bob   agent    (agent:user-2)]  ← CONFLICT_REPORT ✓
```

bug 确认。

## 修

加一个 helper（`mpac_bridge.py`，30 行 + 0 行删）：

```python
def _co_principal_for_owner(principal_id: Optional[str]) -> Optional[str]:
    """Map a principal_id to its sibling for the same human owner.

    Two surfaces represent one human:
      * user:N        — the browser session WS
      * agent:user-N  — the Claude relay WS
    """
    if principal_id.startswith("agent:user-"):
        return f"user:{principal_id[len('agent:user-'):]}"
    if principal_id.startswith("user:"):
        return f"agent:user-{principal_id[len('user:'):]}"
    return None
```

routing 那段改成：

```python
elif resp_type == "CONFLICT_REPORT":
    payload = resp.get("payload", {})
    involved = {payload.get("principal_a"), payload.get("principal_b")}
    involved.discard(None)
    for pid in list(involved):
        involved.add(_co_principal_for_owner(pid))   # ← 新增 4 行
    involved.discard(None)
    for pid in involved:
        await _send_to(session, pid, resp)
```

`_send_to` 对没连接的 pid 是 no-op，所以 user↔user 老路径（pure browser）也不会回归 — 加进去的 sibling 没人接，等于啥也没发生。

## 产品感：Routing 决策是否对称要看真实主体
