## 设计决策（20 分钟拍完）

### 扩展 Scope，不改协议文档

`Scope.extensions` 里新增两个可选字段（都挂在既有的 escape hatch 下，SPEC.md 继续不动）：

```python
Scope(
    resources=["utils.py"],
    extensions={
        # 0.2.1 原字段，保留：
        "impact": ["main.py", "api.py"],

        # 0.2.2 新增：
        "impact_symbols": {
            "main.py": ["utils.foo"],   # scanner: main.py 只用 utils.foo
            "api.py":  ["utils.bar"],   # scanner: api.py 只用 utils.bar
            "docs.py": None,            # `import utils` → 通配，无法细化
        },
        "affects_symbols": ["utils.foo"],   # agent 声明：我只动 foo
    }
)
```

### 冲突判定：双边都有信息才精细化，否则退回文件级

```
对 Alice.impact ∩ Bob.resources 里的每个文件 f：
    如果 Alice 没声明 affects_symbols → 文件级：冲突（保守）
    如果 scanner 把 f 的 impact_symbols 标成 None（通配） → 冲突（保守）
    否则 affects_symbols ∩ impact_symbols[f] 非空 → 冲突
    否则 → 不冲突 ← 新增的精度收益
对称地再跑一次 Bob → Alice
```

**关键**：0.2.2 **只移除**误报，**永不制造**新的漏报 —— 任何 0.2.1 会报的冲突，0.2.2 至多照样报。

### 符号命名：fully-qualified

`utils.py` 里的 `foo` → `"utils.foo"`；`pkg/sub.py` 里的 `Bar` → `"pkg.sub.Bar"`。Scanner 遇到 `from utils import foo as f` 记录**原名** `utils.foo`（agent 关心的是自己的 `foo` 被谁用，和别人起了什么别名无关）。

### 通配语义 —— 何时退回文件级

Scanner 对以下两种情况返回 `None`，强制退回文件级：
- `import utils`（没 `from ... import ...`，下游可能用任何属性）
- `from utils import *`（星号导入）
- 同文件里既有 `import utils` 也有 `from utils import foo` —— 通配 wins（安全第一）

## 对 MPAC 产品定位的一次刻度校准

昨天 KAIYANG 说的话值得重复一遍：**"我们主要是为了做协调"**。

"协调"里最让用户恼火的不是**漏报**（会出 bug），而是**误报** —— 每次误报都让用户对协议失去一点信任。"你又拦我了？我改的根本就不是人家依赖的东西"。

今天把这条最典型的误报路径修掉，是在给协议本身"加权威"：**MPAC 报出来的冲突更值得认真看**。这比加新功能（比如"能不能跑代码"）对用户体验的影响大得多。

接下来的方向（按 backlog 优先级）：
1. **Attribute-chain 解析**（把通配从 "import X" 这种常见写法里解放出来）
2. **内嵌 Claude agent 声明 symbols**（让线上"人 + Claude"的 demo 也享受精度）
3. **前端 UI 在冲突报告里显示具体符号**（让人**看得懂**冲突原因）

这三条连起来，"协调"这条主线又能深一档。

1. 浏览器 DevTools → Network → WS → `INTENT_ANNOUNCE`，看 payload `scope.extensions.impact_symbols` 的内容。如果 Bob 改的那个文件**不在**Alice 的 `impact` 里 → scanner 根本没把它识别成依赖（可能是 `import` 写法特殊，或项目里有非标准包结构）

3. 如果应该报却没报，查 `_extract_imports_detailed` 是不是漏解析了某种 AST 节点 —— 加个回归测试
