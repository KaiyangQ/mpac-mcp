## 根因排查（两个叠加问题）

我让他跑了一个自查脚本（把 python3 版本、pip 版本、claude 位置、网络全打出来），立刻看清两个**都是真 bug**的问题：

### 问题 1：macOS Command Line Tools 的 Python 太老

```
python3 → /usr/bin/python3 → 3.9.6
```

但 **`mpac-mcp` 的依赖 `mcp` 要求 Python ≥ 3.10**。所以 `python3 -m pip install mpac-mcp` 在他这台机器上**根本装不上**：

```
ERROR: Could not find a version that satisfies the requirement mpac-mcp>=0.2.2
(from versions: none)
Requires-Python >=3.10
```

他其实**有** `/opt/homebrew/bin/python3.12`（brew 装的），但脚本盲用 `python3` 这个名字，先撞到系统那个。

### 问题 2：pip 21.2.4 不认识 `--break-system-packages`

Command Line Tools 捆绑的 pip 年纪巨大。脚本的三档 fallback：

1. `pip install` → PEP 668 拦
2. `pip install --user` → PEP 668 也拦
3. `pip install --break-system-packages` → **pip 不认识这个参数** → usage 信息满屏

最后那条错误信息让人误以为是 pipx 的事，其实根本是 pip 版本。

## 修法：`222d444` bootstrap scripts scan for Python >= 3.10 + auto-upgrade old pip

两处同时改 —— `.sh` 和 `.ps1`：

**Python 选择改成扫描 + 版本门槛**：

```bash
PYTHON=""
for cand in python3.13 python3.12 python3.11 python3.10; do
    if command -v "$cand" >/dev/null 2>&1; then
        PYTHON="$cand"
        break
    fi
done
# Fallback: 只有 python3 本身 ≥ 3.10 才用它
if [ -z "$PYTHON" ] && command -v python3 >/dev/null 2>&1; then
    if python3 -c 'import sys; sys.exit(0 if sys.version_info >= (3,10) else 1)'; then
        PYTHON="python3"
    fi
fi
if [ -z "$PYTHON" ]; then
    die "No Python >= 3.10 found. macOS: brew install python@3.12  ..."
fi
```

**Pip 自动升级**：

```bash
PIP_MAJOR=$($PYTHON -c "import pip; print(pip.__version__.split('.')[0])" 2>/dev/null || echo 0)
if [ "$PIP_MAJOR" -lt 23 ]; then
    say "Upgrading pip (current < 23 doesn't know --break-system-packages)..."
    $PYTHON -m pip install -q -U pip 2>/dev/null \
        || $PYTHON -m pip install -q -U --user pip 2>/dev/null \
        || true
fi
```

**PATH 加固**：python 3.12 装的 `mpac-mcp-relay` 落在 `/opt/homebrew/bin/`（brew prefix），绝大多数人 PATH 里有；但为了保险，脚本显式把 `$(command -v $PYTHON)` 的 dirname prepend 到 PATH。

PowerShell 版本同步改，加了 `py launcher` 相关的考虑（Windows 上 `python3.12` 常常不存在，只有 `python.exe`）。

# 事件二：Files 面板想要右键

## `5db3539` right-click in Files panel to add/delete — matches VSCode habits

**新组件 `components/file-context-menu.tsx`** —— 不走 Radix 的 `ContextMenu` primitive，因为那个要求每个 target wrap 一次，我们是递归树 + 还要覆盖空白区域，一个统一的 portal + cursor-positioned menu 更合适。

功能：

| 在哪儿右键 | 弹出菜单 |
|---|---|
| 空白区域 | New file |
| 文件行（`main.py`） | New file + Delete `main.py`（红） |
| 文件夹行（`pkg/`） | New file in `pkg/`（选了之后 modal 预填 `pkg/`） |

附属：`NewFileModal` 加 `initialPath` prop；empty-state 文案从 "Click + to add one" 改成 "Click + or right-click here to add one" —— 给还没发现右键的人一个 hint。

## 技术小细节

- 菜单用 `createPortal` 挂到 `document.body`，`position: fixed`，viewport clamp（右下角点不会截掉）
- Click-outside / `Escape` 自动关
- 行级 `onContextMenu` 的 `stopPropagation` 很关键 —— 不然右键一个文件会**同时**触发外层"空白区域"的 handler，把 target 覆盖成 undefined
- 递归 FileTree 把 `onContextMenu` prop 往下传，每一层都支持

# 下午加演：给内测剧本 review 出来的 scenario 4 静默失败兜底（`mpac 0.2.4`）

下午回过头审 `docs/BETA_SCENARIOS.md`，特别是 scenario 4 那条"**0.2.3 attr-chain 补丁**"。直觉觉得不对 —— seed 里 `service.py` 写的是 `from scenario_4_attr_chain import cache` + `cache.store(...)`，不是 0.2.3 那次明确验证过的 `import utils; utils.foo()` 形态。跑了一个 5 行 Python 对 0.2.3 的 scanner 实测，**得到的不是"退回文件级"，是 `impact_map = {}` —— scanner 连 service.py 是 importer 都不承认**。

## 根因

`mpac_protocol/analysis/import_index.py` 的 `_extract_imports_detailed`：

- `ast.Import` 分支会往 `alias_to_module` 里填（v0.2.3 的 attr-chain resolver 唯一输入）
- `ast.ImportFrom` 分支**只** `out.append((module, symbols))`，**不填** `alias_to_module`
- 所以 `from scenario_4_attr_chain import cache` 的 `cache` 永远不会被 attr-chain walker 看到
- 紧接着 match loop 的两条 `resolved == tmod` / `resolved.startswith(tmod + ".")` 覆盖不到"imported 模块是 target 的**父**包"这个方向 → service.py 连 wildcard importer 都没被标上 → 零冲突

这不是小坑。`from pkg import mod` + `mod.attr()` 是真实 Python 项目里**最常见**的写法（PEP 8 / black / isort 都默认 absolute import），0.2.3 那句"覆盖最常见 Python import 写法"其实没兑现。

## 修法：`e45c920` mpac 0.2.4 (scanner only): resolve from-pkg-import submodule + attr-chain

`ast.ImportFrom` 分支**额外**推测性注册 `alias_to_module[local_name] = f"{mod}.{a.name}"`，让 attr-chain walker 有机会走一遍 `mod.attr()`。为防止推测污染原有路径，给 emit tuple 加第三元 `is_speculative: bool`：

- `is_speculative=True` 的 entry 在 match loop 里**只能走 exact-module 相等这条**，**禁止**走 "submodule of target package" 的 wildcard 分支 —— 否则 target 是 `pkg/__init__.py` 时会被吸走，覆盖 legacy `(pkg, [mod])` 算出来的精确符号
- 推测的 alias 如果被裸引用污染（`ref = cache` + `cache.store()` 混用）→ **silent drop**，不 emit wildcard，和 pre-0.2.4 行为一致（防 submodule-absorption 陷阱）
- 不动 SPEC.md / 不动匹配规则 / 不动协议

## 新增 4 个测试 + 实测过清单

单测：
