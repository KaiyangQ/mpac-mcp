# Daily Report — 2026-04-06

## 概述

今天在 MPAC v0.1.12 已有的代码编辑分布式验证基础上，设计并实现了一个全新的消费级多主体协调场景：**家庭出游行程规划**。三个独立的 AI agent（爸爸、妈妈、12岁小孩）各自调用 Claude API，通过 MPAC 协议在 WebSocket 上协调一次 5 天家庭自驾游。这是协议首次在非技术领域、三方（而非两方）场景下进行端到端验证。运行结果：24 条 MPAC 消息、3 个冲突被自动检测并通过自然语言协商解决、12 次独立的 Claude API 调用、最终生成完整行程（预估费用 ¥6,680 / ¥15,000 预算）。

---

## 完成事项

### 1. 用例设计文档

新建 `version_history/v0.1.12_conformance_closure/MPAC_v0.1.12_Family_Trip_Use_Case.md`，详细设计了家庭出游场景的 MPAC 建模方式：

- **三个主体（principal）**：爸爸（预算控制+司机+户外爱好者）、妈妈（餐饮住宿+文化体验+海鲜过敏）、小孩（主题乐园+水上活动+自由时间）
- **共享状态建模**：`task_set` scope，资源包括 `itinerary://day-1` 到 `itinerary://day-5` 以及 `budget://accommodation` 等 5 个预算类别
- **权限模型**：父母 = `owner`（可提议+可决策），小孩 = `contributor`（只可提议）
- **执行模型选择**：`pre_commit`（行程变更需 coordinator 授权后才能生效），防止单个 agent 单方面订酒店
- **完整消息流设计**：从 HELLO 握手到 INTENT_ANNOUNCE、CONFLICT_REPORT、CONFLICT_ACK、RESOLUTION、OP_BATCH_COMMIT、SESSION_CLOSE 的全生命周期
- **5 个边界场景**（what-if）：营地不合格时的降级、已提交行程的 OP_SUPERSEDE、断连恢复 INTENT_CLAIM、预算超支回滚、第二个家庭中途加入

文档解释了为什么这个场景需要 MPAC 而不是 A2A：三个家庭成员是真正的独立主体，各有不可让渡的约束（妈妈的海鲜过敏、爸爸的预算控制），没有单一 orchestrator 拥有完全权限。

### 2. 代码实现 — 三 Agent 家庭旅行 Demo

在 `ref-impl/demo/distributed/` 目录下新增两个文件：

**`trip_agent.py` — 旅行规划 Agent：**

- 每个实例有独立的 Claude system prompt，编码主体的偏好、约束和职责
- 三种 Claude API 调用接口：
  - `decide_intent(trip_context, other_intents)` → 返回 intent JSON（intent_id、objective、scope_resources、assumptions、priority）
  - `respond_to_conflict(conflict_info, own_intent, other_intent)` → 返回协商立场 JSON（ack_type、position、flexibility、compromise_proposal）
  - `generate_itinerary(day_numbers, resolution_context, other_plans, budget_info)` → 返回详细行程 JSON（每天的 morning/afternoon/evening/accommodation/meals/cost/notes）
- WebSocket 通信层：connect、send、recv、drain_inbox
- MPAC 消息辅助方法：do_hello、do_announce_intent、do_conflict_ack、do_commit、do_batch_commit、do_goodbye

**`run_family_trip.py` — 9 阶段编排器：**

- Phase 0: 创建 3 个 TripAgent（Dad=owner, Mom=owner, Kid=contributor）
- Phase 1: 启动 WebSocket coordinator
- Phase 2: 3 个 agent 依次 HELLO 加入 session
- Phase 3: 3 个 agent **并发**调用 Claude API 决定 intent（5.9 秒完成 3 次 LLM 调用）
- Phase 4: INTENT_ANNOUNCE → coordinator 自动检测 scope overlap → 3 个 CONFLICT_REPORT
- Phase 5: 每个冲突的双方 agent **并发**调用 Claude API 表达协商立场 → coordinator 以 `merged` 决策解决
- Phase 6: 3 个 agent **并发**调用 Claude API 生成详细行程（17.6 秒完成 3 次 LLM 调用）
- Phase 7: 通过 OP_BATCH_COMMIT 提交行程（每个 agent 提交各自负责的天数，使用 `all_or_nothing` 原子语义）
- Phase 8: 展示最终合并行程 + 总预算
- Phase 9: GOODBYE → SESSION_CLOSE

**`ItineraryStore` — 内存共享状态：**

- 跟踪每天行程的 `state_ref`（SHA-256 hash of plan JSON）
- 模拟真实的共享文档状态管理

### 3. Scope 修复

运行中发现 `task_set` scope overlap 未被检测到。根因：`trip_agent.py` 创建 Scope 时将资源放入 `resources` 字段，而 `scope.py` 的 overlap 检测对 `task_set` 使用 `task_ids` 字段。修复：`do_announce_intent` 中改为 `Scope(kind="task_set", task_ids=resources)`。这是 `task_set` scope kind 首次在真实 demo 中使用，暴露了一个 API 层面的使用陷阱——Scope 模型的字段名与 scope kind 绑定，但没有构造时校验。

### 4. 实际运行结果

Demo 端到端成功运行。关键指标：

| 指标 | 值 |
|------|-----|
| MPAC 消息交换 | 24 条 |
| 参与者 | 3 个（Dad, Mom, Kid）|
| Intent 宣告 | 3 个（各 agent 1 个）|
| 冲突检测 | 3 个（Dad-Mom, Dad-Kid, Kid-Mom）|
| 冲突解决 | 3/3 全部 MERGED |
| Operation 提交 | 7 个（通过 3 次 OP_BATCH_COMMIT）|
| Claude API 调用 | 12 次（3 intent + 6 position + 3 itinerary）|
| 总耗时 | ~1 分钟 |
| 行程预估费用 | ¥6,680 / ¥15,000 预算 |

**3 个冲突的协商过程：**

1. **Dad vs Mom（Day 2-3 重叠）**：Dad 想露营+徒步，Mom 想住民宿+文化体验。双方各自 `disputed`（flexibility=medium），均提出"一晚露营+其余住民宿"的妥协。Coordinator 以 `merged` 解决。

2. **Dad vs Kid（Day 2 重叠）**：Kid 想去 Hello Kitty 乐园，Dad 安排了户外活动。Kid 的 agent 主动 `accepted`（flexibility=high），表示愿意把乐园挪到其他天。展示了合作式 agent 行为。

3. **Kid vs Mom（Day 2, Day 4 重叠）**：Mom 的 agent `accepted`（flexibility=high），愿意调整文化活动配合 Kid 的乐园计划。展示了家庭场景中的自然让步。

Session health 在冲突期间从 `healthy` → `degraded`（3 个 open conflicts），解决后恢复 `healthy`。
