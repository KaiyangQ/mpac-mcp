# 上午：结构性怀疑两条都用数据排除

## Q1：100 agent 这个协议还能用吗？

昨天刚把 `dependency_breakage` 做完 —— `compute_scope_impact` 每次 announce 都要把整个项目的 `.py` 文件全 parse 一遍。直觉上这就是**规模化**时最先崩的地方。加上 coordinator 是单进程 Python（GIL）、`_detect_scope_overlaps` 是 O(N) over live intents —— 三个猜测的瓶颈叠加，不压一下不知道真相。

### 测试设计

写了 `scripts/mpac_load_test.py`，**在进程内**直接起 `SessionCoordinator`（不走 HTTP / 不走 SQLite / 不走 Claude）—— 隔离出协议本身的性能曲线。

合成一个 200 文件的 Python 项目（30% 是"hub"模块被多人 import），模拟 N 个 agent 做 `--interleave` 工作流：所有人先 announce，再所有人 withdraw，周期 5 次。Interleave 模式让 live intent 集合顶在 N，这是**最坏情况**（现实里 agent 是 announce-做事-withdraw 串行的，live 集合远小于 N）。

### 结果汇总

| N agents | 项目文件 | Wall | 吞吐 | P50 announce | P99 | Analyzer P50 | Coord P50 | RSS 峰值 |
|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| 10  | 200 | 0.12s | **418/s** | 2.3ms | 2.7ms  | 2.2ms | 0.1ms | 62 MB |
| 50  | 200 | 0.74s | 338/s | 2.7ms | 3.4ms  | 2.3ms | 0.5ms | 63 MB |
| **100** | 200 | 2.0s | **250/s** | **3.3ms** | **4.8ms** | 2.3ms | 1.0ms | 68 MB |
| 200 | 200 | 7.3s  | 137/s | 4.8ms | 8.7ms  | 2.4ms | 2.4ms | 85 MB |
| 500 | 200 | 71s   | 35/s  | 11.8ms| 25.7ms | 2.5ms | 9.3ms | 211 MB |
| 100 | **500** | 3.9s | 127/s | 7.4ms | 10.2ms | 6.2ms | 1.1ms | 66 MB |

### 三个关键发现

**1. 两个瓶颈**独立增长**，这是架构上的好事**

- **Analyzer cost** 只跟**项目文件数**有关（200 文件 ~2.3ms，500 文件 ~6.2ms —— 线性）
- **Coordinator cost** 只跟**live intents 数**有关（10 ~0.1ms → 100 ~1ms → 500 ~9ms —— 线性）

它俩不交叉爆炸。优化各自有独立的杠杆点。

**2. 100 agents 同 session，P99 延迟 4.8ms，吞吐 250 announces/秒**

合理假设：每个 agent 每分钟 announce 一次 → 峰值 ~1.7 announces/秒
实测吞吐：250/秒
→ **150× 冗余**。头都没抬。

**3. 真正变慢的拐点在 N ≥ 300 live intents**

N=500 时 P99 跳到 26ms、最大 98ms。但注意这是"500 个 intent 同时 live"的极端情况 —— 正常使用 live 集合 ≈ **并发干活人数**，不是总 agent 数。

内存不是问题（500 agent + 500 live intent 只多用 150MB）。

### Claude 订阅问题顺手回答

做 load test **不需要真 Claude**：
- Claude Pro 单用户订阅 + rate limit → 撑不起并发 100
- 真 agent 响应 2-10 秒 / 请求 → 远大于 MPAC 3ms 的协议延迟 → **Claude 本身就是瓶颈**，压力到不了 MPAC
- 多 agent paper 的标准做法就是 1-2 个真 agent + N-2 个模拟 envelope 填充

# 下午：mpac 0.2.2 符号级精度上线

## 开工前已知的短板

0.2.1 的 `dependency_breakage` 是文件级：Alice 改 `utils.py` 里的 `foo`，Bob 改 `api.py`（只 `from utils import bar`），两人根本不碰同一个符号，0.2.1 **会报冲突**。这是典型误报。

## 设计决策（20 分钟拍完）

### 扩展 Scope，不改协议文档

`Scope.extensions` 里新增两个可选字段（都挂在既有的 escape hatch 下，SPEC.md 继续不动）：

```python
Scope(
    resources=["utils.py"],
    extensions={
        # 0.2.1 原字段，保留：
        "impact": ["main.py", "api.py"],

        # 0.2.2 新增：
        "impact_symbols": {
            "main.py": ["utils.foo"],
            "api.py":  ["utils.bar"],
            "docs.py": None,            # `import utils` → 通配
        },
        "affects_symbols": ["utils.foo"],   # agent 声明：我只动 foo
    }
)
```

### 冲突判定：双边都有信息才精细化，否则退回文件级

```
对 Alice.impact ∩ Bob.resources 里的每个文件 f：
    如果 Alice 没声明 affects_symbols → 文件级：冲突（保守）
    如果 scanner 把 f 的 impact_symbols 标成 None（通配） → 冲突（保守）
    否则 affects_symbols ∩ impact_symbols[f] 非空 → 冲突
    否则 → 不冲突 ← 新增的精度收益
对称地再跑一次 Bob → Alice
```

**关键**：0.2.2 **只移除**误报，**永不制造**新的漏报 —— 任何 0.2.1 会报的冲突，0.2.2 至多照样报。

# 晚上：mpac 0.2.3 attr-chain + 前端具体符号 UI

## 0.2.2 遗留的两个短板

下午刚上线 0.2.2 之后，立刻看到两个扎眼的残留问题：

1. **Attribute-chain 没处理**：0.2.2 里 `import utils` + `utils.foo()` 这种**最常见的 Python import 写法**会被扫描器标通配 → 退回文件级 → 误报回来了。0.2.2 的符号精度在这种情况下**完全没生效**。

2. **前端看不见具体符号**：后端已经算出"是哪个符号在撞"（`impact_symbols ∩ affects_symbols`），但 `CONFLICT_REPORT` 只传了笼统 `category`。用户看到 "Dependency conflict" 但不知道具体什么冲突 —— 白白浪费后端已经有的信息。

## 改动：两块，一个 commit

### `5e0ebc7` mpac 0.2.3: attribute-chain resolution + dependency_detail

#### Part A：扫描器 attribute-chain 解析

对每个 `import X` / `import X as Y` 绑定，遍历 AST 收集所有 `Attribute(value=Name(alias))` 节点，把访问过的属性名搜集起来。

**什么时候退回通配**：
- 任一 `Name(alias)` 节点的 parent **不是** `Attribute.value`（比如 `x = utils`、`return utils`、`isinstance(m, utils)`）→ 该 alias 通配
- `import pkg.sub` 这种带点的 → 通配（区分"submodule vs attribute"需要模块图）
- 没有任何使用（`import utils` 然后没碰它）→ 通配（防动态 eval / getattr）

**实现亮点**：
- 临时在节点上挂 `_mpac_parent` 属性反查父节点
- 每个 alias 独立追踪
- 嵌套作用域也算（函数体内的 `utils.foo` 照样被收集）

**8 个新测试**：
- `import utils + utils.foo() + utils.bar()` → `['utils.foo', 'utils.bar']`
- `import utils as u + u.foo()` → `['utils.foo']`
- `import utils + x = utils + utils.foo()` → `None`（bare use 一票否决）
- `import utils + from utils import bar + utils.foo() + bar()` → `['utils.bar', 'utils.foo']`
- `import pkg.sub` → `None`（保守）
- `import utils` 无使用 → `None`
- `utils.Class.method()` → `['utils.Class']`（记外层 attribute）
- 函数体里的 `utils.foo()` → `['utils.foo']`

#### Part B：CONFLICT_REPORT payload 带 dependency_detail

新增 `mpac_protocol.core.scope.compute_dependency_detail(a, b)`：

```python
{
  "ab": [{"file": "<b's file>", "symbols": [...] | None}, ...],
  "ba": [{"file": "<a's file>", "symbols": [...] | None}, ...]
}
```

- `ab` = a 的改动波及 b 的哪些文件（和 `principal_a` / `principal_b` 对齐）
- 每个文件的 `symbols` = **交集** `affects_symbols ∩ impact_symbols[file]`（精确）
- 或 `None`（退回文件级）

`coordinator._detect_scope_overlaps` 只在 `category == "dependency_breakage"` 时带上这个字段。

**5 个新测试 + 1 coord 断言**。

### `62a85b2` web-app: render clashing symbols in conflict card

前端三层改动：

1. **`envelope-types.ts`**：新增 `DependencyDetailEntry` + `DependencyDetail` 类型；`ConflictReportPayload.dependency_detail?`
2. **`mpac-session.ts`**：`LiveConflict` 透传 `dependency_detail`
3. **`projects/[id]/page.tsx` 的 `ConflictCard`**：把 `ab/ba` 两个方向摊平成 `{editor, consumer, file, symbols}` 行，渲染：
   - 符号精确时：**"Alice is changing `utils.foo` — affects Bob's `main.py`"**
   - 文件级回退时：**"Alice is editing a file imported by — affects Bob's `main.py`"**
   - 标签从 `dependency_breakage` 改成 `Dependency`（可读性）

### `a2ae125` web-app: require mpac>=0.2.3

`requirements.txt` 0.2.2 → 0.2.3。

## 0.2.3 生产验证

关键 case：**`main.py` 里是 `import utils` + `utils.foo()`** —— 0.2.2 会标通配，0.2.3 必须解析成 `['utils.foo']`。

```
S1. Alice 声明 symbols=[utils.foo]
  ✅ attr-chain resolved: impact_symbols[main.py]==['utils.foo']   ← 生产端真的解析了
S2. Bob announces main.py
  ✅ category = dependency_breakage
  ✅ dependency_detail present in payload                           ← 前端拿到了
  ✅ ba: [{"file":"main.py","symbols":["utils.foo"]}]               ← 符号精确到位
```
