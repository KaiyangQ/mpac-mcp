## Q1 回答：Load Test 5 档实测

### 测试设计

写了 `scripts/mpac_load_test.py`，**在进程内**直接起 `SessionCoordinator`（不走 HTTP / 不走 SQLite / 不走 Claude）—— 隔离出协议本身的性能曲线。

合成一个 200 文件的 Python 项目（30% 是"hub"模块被多人 import），模拟 N 个 agent 做 `--interleave` 工作流：所有人先 announce，再所有人 withdraw，周期 5 次。Interleave 模式让 live intent 集合顶在 N，这是**最坏情况**（现实里 agent 是 announce-做事-withdraw 串行的，live 集合远小于 N）。

### 结果汇总

| N agents | 项目文件 | Wall | 吞吐 | P50 announce | P99 | Analyzer P50 | Coord P50 | RSS 峰值 |
|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| 10  | 200 | 0.12s | **418/s** | 2.3ms | 2.7ms  | 2.2ms | 0.1ms | 62 MB |
| 50  | 200 | 0.74s | 338/s | 2.7ms | 3.4ms  | 2.3ms | 0.5ms | 63 MB |
| **100** | 200 | 2.0s | **250/s** | **3.3ms** | **4.8ms** | 2.3ms | 1.0ms | 68 MB |
| 200 | 200 | 7.3s  | 137/s | 4.8ms | 8.7ms  | 2.4ms | 2.4ms | 85 MB |
| 500 | 200 | 71s   | 35/s  | 11.8ms| 25.7ms | 2.5ms | 9.3ms | 211 MB |
| 100 | **500** | 3.9s | 127/s | 7.4ms | 10.2ms | 6.2ms | 1.1ms | 66 MB |

### 三个关键发现

**1. 两个瓶颈互相**独立**，这是架构上的好事**

- **Analyzer cost** 只跟**项目文件数**有关（200 文件 ~2.3ms，500 文件 ~6.2ms —— 线性）
- **Coordinator cost** 只跟**live intents 数**有关（10 ~0.1ms → 100 ~1ms → 500 ~9ms —— 线性）

它俩不交叉爆炸。优化各自有独立的杠杆点。

**2. 100 agents 在同一 session，P99 延迟 4.8ms，吞吐 250 announces/秒**

合理假设：每个 agent 每分钟 announce 一次 → 峰值 ~1.7 announces/秒
实测吞吐：250/秒
→ **150× 冗余**。根本还没开始喘气。

**3. 真正变慢的拐点在 N ≥ 300 live intents**

N=500 时 P99 跳到 26ms、最大 98ms。但注意这是"500 个 intent 同时 live"的极端情况 —— 正常使用 live 集合 ≈ **并发干活人数**，不是总 agent 数。

### 内存不是问题

500 agent + 500 live intent 只多用 150MB。5000 agent 的量级都扛得住。

### Claude 订阅那条也顺手回答了

KAIYANG 顺口问了「能都调用 Claude 订阅吗」—— 做 load test **根本不需要真 Claude**：
- Claude Pro 单用户订阅 + rate limit → 撑不起并发 100
- 真 agent 响应 2-10 秒 / 请求 → 远大于 MPAC 3ms 的协议延迟 → **Claude 本身就是瓶颈**，压力到不了 MPAC
- 多 agent paper 的标准做法就是 1-2 个真 agent + N-2 个模拟 envelope 填充

### 什么时候需要优化（现在不用）

如果哪天真的要撑 500+ live intents，按性价比排序的三件事：

1. **增量 import graph**：缓存反向依赖图，file write 时增量更新 —— analyzer 成本 2-6ms → <0.5ms
2. **冲突检测倒排索引**：当前 O(N) 扫所有 intent；改成"文件 → 谁在 claim"的倒排，O(1) 查询
3. **Coordinator async 化 + per-session lock**：不同 session 真正并行

但**现在不用做**。有 150× 冗余的情况下去优化是过度工程。

## Q2 回答：mpac-mcp 实测 token 成本

### 方法

用 `tiktoken`（cl100k_base 编码，和 Claude tokenizer 同族，误差 <5%）对 `mpac-mcp` 的 6 个 tool 做序列化 + 计数：

```python
from mpac_mcp.relay_tools import mcp
tools = await mcp.list_tools()
# 序列化成 MCP 实际发给 Claude 的 JSON 形式
# 用 tiktoken 计数
```

### 结果

| 工具 | 字符 | tokens |
|---|---:|---:|
| `list_project_files` | 204 | 50 |
| `read_project_file` | 280 | 74 |
| `write_project_file` | 679 | 166 |
| `check_overlap` | 609 | 147 |
| `announce_intent` | 692 | 175 |
| `withdraw_intent` | 490 | 121 |
| **Total（6 tools）** | **2954** | **733** |
| + MCP preamble | | ~18 |
| **最终开销** | | **~751 tokens** |

### 对比作者的指控

| | tokens | 占 200K 上下文 |
|---|---:|---:|
| 作者说的"平均 MCP" | ~10,000 | 5% |
| 作者说的"5 MCP 噩梦组合" | ~50,000 | 25% |
| **mpac-mcp 实测** | **~751** | **0.38%** |

**比作者吐槽的平均 MCP 小 13 倍，比"噩梦组合"小 67 倍。**

### 为什么我们这么省

对比 GitHub MCP（一口气塞 50+ 个 tools：list repos / create PR / merge / comment / ...），我们只有 **6 个窄接口**：

- 3 个文件操作：`list / read / write`
- 3 个协调语义：`announce / check / withdraw`

每个 tool 的 docstring 都短得能看得完。没塞 README、没塞 API 文档、没塞示例。

### 按作者建议"瘦身"是否值得

如果真把文件操作搬去 CLI（`mpac-cli`），只留协调 tool：

| | 留下 | tokens |
|---|---|---:|
| 当前 | 6 个全在 MCP | 751 |
| 瘦身后 | 只留 `announce / check / withdraw` | 461 |
| 节省 | | **-290 (0.15% of context)** |

**不值得**。为 0.15% 的 context 做架构分叉，ROI 太低。要分的话得是因为 CLI 脚本化有别的独立价值。

### 作者的 3 层框架其实**认可 MPAC**

| 他的层 | 用途 | 我们对应 |
|---|---|---|
| Skills（静态 markdown） | "The What"：规范 | **不适用** —— MPAC 是运行时有状态的 |
| CLI wrappers | "The How"：一次性动作 | 文件 ops **可以**搬去，协调 tool 不行 |
| MCP | **真正的协议边界 / 动态发现 / 跨信任域** | ✅ `announce/check/withdraw` 正是这类 |

按他**自己**的分类，MPAC 的核心协调 tools 就是 "MCP 应该存在的理由"：
- ✅ 跨信任域（不同 principal）
- ✅ 动态状态（live intents）
- ✅ 本身就是协议边界

## 今天 MPAC 的产品定位更清楚了

> **"我们主要是为了做协调。"**
> ——— KAIYANG 今天说的

这句话值得记下来。之前讨论过"能不能让 Claude 帮我们跑代码"、"能不能加 sandbox"、"能不能做运行期冲突检测"—— 这些都是**扩展**，但不是**定位**。MPAC 的定位就是**协调**（coordination）。

协调里面还有很多路没走完：
- **符号级精细化**（A 改 `utils.foo`，B 只 import `utils.bar` → 目前会误报）
- **多语言**（scanner 接口源无关，加 JS/TS/Go parser 就扩）
- **传递闭包**（一层反向依赖够不够用）
- **语义协调超过 import**（API 契约、共享数据模型、config 键）
- **Intent 粒度**（文件 → 函数 → 行区间）
- **授权模型**（governance / 谁能接管）

明天该挑一个往深打，不是再扩横向能力。
