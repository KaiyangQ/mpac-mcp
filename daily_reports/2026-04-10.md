### 2. Bug 修复：`_generate_fix` 输出解析（P0）

**问题描述**：

场景 2 中，fangxm 的 agent 在 `task Fix the bug in auth.py` 时，Claude API 返回了带 markdown 解释文字的代码：

```
Looking at the code, I can identify a bug in the `validate_token` function...

```python
"""Authentication module — has several known bugs for testing."""
import hashlib
...
```​
```

旧的解析逻辑只检查 `result.startswith("```")`，不以 ``` 开头就原样使用。结果**整段文本（解释 + 代码块标记 + 代码）被当作 Python 文件内容提交**。

**根因**：prompt 里写了 `"Return ONLY the complete fixed file — no markdown fences, no explanations."`，但 LLM 不总是遵守格式指令。旧代码缺少防御性解析。

**修复**：新增 `_extract_code()` 方法，用正则搜索所有 ` ```python...``` ` 代码块，取最长的一个作为文件内容。如果没找到代码块才 fallback 到旧的首尾 strip 逻辑。

**改动文件**：
- `mpac-package/src/mpac_protocol/agent.py` — 新增 `_extract_code()` 静态方法替代旧的 4 行 strip 逻辑
- `ref-impl/demo/distributed/run_e2e.py` — 同样的修复

**设计原则**：不信任 LLM 一定遵守格式指令，用代码兜底。

### 3. Bug 修复：server 未广播成功的 OP_COMMIT/INTENT_ANNOUNCE（P1）

**问题描述**：

场景 3 测试中发现，Bob 看不到 Alice 的 commit 通知（没有 `>> alice committed changes to: auth.py` 的异步消息）。

**根因分析**：

问题的调用链路：

```
Agent A 发 OP_COMMIT
  → coordinator.process_message() 返回 []（成功，无协议响应）
  → server 路由循环无消息可广播
  → server._apply_file_changes() 广播 FILE_UPDATED（sideband，用 "type" 而非 "message_type"）
  → Agent B 的 _listen() 把它路由到 sideband_inbox
  → _print_event() 从未被调用
  → Bob 看不到通知
```

coordinator 返回 `[]` 是**正确的**——它的职责是处理状态机逻辑（冲突检测、stale ref 校验等），不负责消息转发。问题在 server 层：server 只转发了 coordinator 生成的响应，**漏掉了"把原始入站消息 relay 给其他参与者"这个职责**。

**修复**：server 检测到成功的 `OP_COMMIT`、`INTENT_ANNOUNCE`、`INTENT_WITHDRAW` 后，主动广播原始消息给所有 agent（OP_COMMIT 广播时剥离 `file_changes` 避免传输大文件内容）。

**安全性验证**：
- 发送者自己也会收到广播，但 `_print_event` 里的 `sender == self.principal_id` 过滤会挡住，不会重复打印
- `_print_event` 在 `_listen` 里是先打印再入队（`_print_event(data)` → `protocol_inbox.put(data)`），所以即使消息后来被 `_do_commit` 的 drain 消费掉，通知已经打印过了
- 广播消息进入 `protocol_inbox` 不会干扰正常流程——`_do_commit` 只检查 `PROTOCOL_ERROR` 和 `CONFLICT_REPORT`，其他消息被 drain 时静默丢弃

**改动文件**：
- `mpac-package/src/mpac_protocol/server.py` — 在消息路由逻辑前新增广播判断

**关键认识**：这不是协议问题，也不是 coordinator 问题。协议定义了消息语义，coordinator 正确处理了状态转换，但 server 作为消息路由器没有履行 relay 职责。三层职责清晰：

| 层 | 职责 | 状态 |
|---|------|------|
| 协议 (SPEC) | 定义消息类型和语义 | 无需修改 |
| Coordinator | 状态机逻辑、冲突检测 | 行为正确 |
| Server | 消息路由和转发 | ← 问题在这里，已修复 |

### 5. 讨论：Agent 通用化——从"只能改代码"到"协作做任何事"

在复盘四场景测试时，提出了一个问题：**MPAC 协议只能用来改代码吗？**

答案是**协议本身是内容无关的**（scope 支持 `file_set`、`entity_set`、`task_set`；OP_COMMIT 传的是任意文本内容 + SHA-256 哈希），已有 `run_family_trip.py` demo 证明协议能处理旅游规划这样的非代码场景。

但**当前 pip 包的 agent 实现是写死给 Python 代码用的**。对比两条路径：

| | `run_family_trip.py`（demo） | `mpac-package`（pip 包） |
|---|---|---|
| 场景 | 旅游规划 ✅ | 只能改代码 |
| scope 类型 | `task_set`（`itinerary://day-1`） | `file_set`（`auth.py`） |
| 共享状态 | `ItineraryStore` | `FileStore` |
| 交互方式 | 自动跑完，单机 | 真人交互，跨网络 |

真正绑死在"改代码"上的只有 **agent 层的几段 prompt**，不涉及协议、coordinator 或基础设施：

| 位置 | 现在硬编码的 | 通用化方向 |
|------|-------------|-----------|
| `_generate_fix` system prompt | `"You are given a Python source file. Fix it..."` | 根据 workspace 内容自适应（放 `.py` 就改代码，放 `.md` 就编辑文档） |
| `_generate_fix` user prompt | `"Return the complete fixed Python file."` | `"Return the complete updated file."` |
| `_extract_code` | 只找 ` ```python ` 代码块 | 找任意代码块，或直接返回原文 |
| `_decide_intent` | 假设目标是改文件 bug | 泛化为"分析任务影响哪些资源" |
| `role_description` | `"Security engineer..."` | 用户自定义 |

**核心思路**：把"领域知识"从代码移到 prompt，让 Claude 根据 workspace 内容自动适应。Agent 代码本身不需要知道自己在什么领域——`FileStore` 存的就是"文件名 → 文本内容"，一个 `auth.py` 和一个 `day-1.md` 在存储层没有区别。

**这是一个好信号**：说明协议抽象做对了（SPEC、coordinator、状态机、冲突检测完全不用改），待泛化的只是最上层的 agent prompt。

**已完成**：agent prompt 通用化改动已提交。去掉了所有 Python/代码特定的硬编码，`_generate_fix` 改为 "Update the file according to the objective"，`_extract_code` 支持任意语言标记和纯文本 fallback，`role_description` 改为可选参数。

### 6. Agent 层新增 `execute_task()` + 六场景自动化测试

#### 问题

原有的 `test_e2e.py` 是 run-to-completion 模式——两个 agent 跑一轮 task 就退出。4/9 日报已指出这种模式无法覆盖长连接场景下的 intent 堆积、stale-state-ref rebase 等问题。而真实联调（两个人通过 ngrok 跨网络操作）又需要两个人在线，效率低。

#### 解决方案

**新增 `MPACAgent.execute_task(task)`**：从 `run_task()` 中拆出任务执行逻辑（intent → conflict → fix → commit → withdraw），不包含 HELLO/GOODBYE。agent 保持连接，可连续执行多个 task。`run_task()` 改为调用 `execute_task()`，行为不变。

**新增 `test_e2e_scenarios.py`**：六场景自动化测试，一个人本地运行。两个 agent 全程保持长连接，顺序跑完所有场景：

| 场景 | 描述 | 验证目标 |
|------|------|---------|
| 1. 无冲突 | Alice → auth.py, Bob → api.py | 不同文件无干扰 |
| 2. 冲突检测 | 两人同时 → auth.py | CONFLICT_REPORT 触发 |
| 3. 依赖链 | Alice 先改，Bob 基于 Alice 版本改 | state_ref 链式更新 |
| 4. 冲突 + rebase | 两人同时 → auth.py，都 proceed | STALE_STATE_REF + 自动 rebase |
| 5. 不对称 yield | 关键安全修复 vs 加注释 | 优先级不对称下的自然 yield |
| 6. 强制 mutual yield | mock `_decide_conflict` 强制双方 yield | mutual yield 检测 + 自动重试全链路 |

运行方式：`python test_e2e_scenarios.py`（需要 Anthropic API key，约 30-40 次 Claude 调用）

### 7. Mutual Yield 检测与自动重试

#### 问题

场景 2 测试中发现：**两个 agent 的 Claude 都选择了 yield**，导致冲突双方各自放弃任务，文件谁都没改。每个 agent 只看到自己的决定（"Yielding to other agent. Task cancelled."），不知道对方也 yield 了。

这是一个 UX 问题，不是协议问题——协议正确传递了 INTENT_WITHDRAW，但 agent 实现没有利用这个信息。

#### 根因

`_decide_conflict` 调用 Claude 做仲裁，决策不可控。当两个任务看起来同等重要时，两边的 Claude 可能都"礼让"。Yield 后 agent 立即返回，不检查对方是否也 yield。

#### 解决方案（纯 agent 层，协议零改动）

1. **追踪 INTENT_WITHDRAW**：`_print_event` 新增 `INTENT_WITHDRAW` 处理，同时将收到的 withdraw 记录到 `_recent_withdraws` 列表
2. **Yield 后等待检测**：agent yield 后等待 3 秒，检查是否收到对方的 INTENT_WITHDRAW
3. **自动重试**：如果检测到 mutual yield，自动重试任务，重试时 `_is_retry=True` 强制 proceed（打破对称性）
4. **单次重试限制**：`_is_retry` 标记防止无限循环

**协议层看到的消息序列完全合法**：

```
Agent A: INTENT_ANNOUNCE → INTENT_WITHDRAW → INTENT_ANNOUNCE (retry) → OP_COMMIT
Agent B: INTENT_ANNOUNCE → INTENT_WITHDRAW → INTENT_ANNOUNCE (retry) → OP_COMMIT
```

Coordinator 不知道也不需要知道"这是 mutual yield 重试"——它只看到标准的 intent 宣告和撤回，照常检测 scope 和冲突。

**改动文件**：
- `mpac-package/src/mpac_protocol/agent.py` — `_print_event` 加 INTENT_WITHDRAW、`execute_task` 和 `_run_task_interactive` 加 mutual yield 检测 + 重试
- `test_e2e_scenarios.py` — 新增场景 5（不对称 yield）和场景 6（强制 mutual yield）

#### 场景 6 的完整验证链路

```
Alice: yield → INTENT_WITHDRAW → 等 3s → 检测到 Bob 也 withdraw → 重试 proceed
Bob:   yield → INTENT_WITHDRAW → 等 3s → 检测到 Alice 也 withdraw → 重试 proceed
→ 再次冲突 → 两边都强制 proceed → first-committer-wins → 后者 rebase → 都提交成功
```

**6/6 场景全部 PASS。**

### 8. MPACAgent API 扩展——接通 coordinator 已有能力

#### 问题

coordinator（`core/coordinator.py`）和 participant（`core/participant.py`）早已实现了 pre-commit 执行模型（OP_PROPOSE → authorization → OP_COMMIT）、INTENT_CLAIM（故障恢复）、CONFLICT_ESCALATE（冲突升级到仲裁者）、arbiter RESOLUTION 等完整能力。但 pip 包的 MPACAgent 只暴露了最基本的 announce → commit 流程，无法调用这些高级协议特性。

同时，server 的消息转发列表只包含 `OP_COMMIT`、`INTENT_ANNOUNCE`、`INTENT_WITHDRAW` 三种，coordinator 生成的 `CONFLICT_ACK`、`CONFLICT_ESCALATE`、`RESOLUTION`、`INTENT_UPDATE` 等消息不会被广播给其他 agent。

#### 解决方案

**MPACAgent 新增公开方法**（纯 agent 实现层，协议和 coordinator 零改动）：

| 方法 | 消息类型 | 用途 |
|------|---------|------|
| `do_heartbeat(status)` | HEARTBEAT | 维持 liveness |
| `do_ack_conflict(conflict_id, ack_type)` | CONFLICT_ACK | 确认/争议冲突 |
| `do_update_intent(intent_id, objective, files)` | INTENT_UPDATE | 修改 intent scope |
| `do_propose(intent_id, op_id, target)` | OP_PROPOSE | pre-commit 提案，等待 authorization |
| `propose_and_commit(...)` | OP_PROPOSE + OP_COMMIT | 完整 pre-commit 流程 |
| `do_claim_intent(...)` | INTENT_CLAIM | 接管崩溃 agent 的 suspended intent |
| `do_escalate_conflict(...)` | CONFLICT_ESCALATE | 升级冲突到仲裁者 |
| `do_resolve_conflict(...)` | RESOLUTION | 仲裁者发布裁决 |

**构造函数扩展**：

- 新增 `roles` 参数——支持 `["owner"]`、`["arbiter"]`、`["contributor"]` 等角色
- 新增 `principal_id` 参数——支持自定义身份（如 `"human:arbiter"`）
- 扩展默认 `capabilities` 列表，包含 intent/conflict/governance 全部能力

**Scope 泛化**：

`_do_announce_intent` 现在根据 `scope_kind` 正确填充 Scope 模型的字段：
- `file_set` → `Scope.resources`
- `task_set` → `Scope.task_ids`
- `entity_set` → `Scope.entities`

之前只硬编码 `resources`，导致 `task_set` 的 scope overlap 检测失败（`scope_overlap()` 对 `task_set` 检查的是 `task_ids` 字段，不是 `resources`）。

**Server 消息转发扩展**：

relay 广播列表从 3 种扩展到 7 种：

```python
# Before
"OP_COMMIT", "INTENT_ANNOUNCE", "INTENT_WITHDRAW"

# After
"OP_COMMIT", "INTENT_ANNOUNCE", "INTENT_WITHDRAW",
"INTENT_UPDATE", "CONFLICT_ACK", "CONFLICT_ESCALATE", "RESOLUTION"
```

**改动文件**：
- `mpac-package/src/mpac_protocol/agent.py` — 构造函数扩展 + 8 个新方法 + scope 泛化
- `mpac-package/src/mpac_protocol/server.py` — relay 广播列表扩展（1 行）

**关键设计原则**：所有改动均为 agent/server 实现层。SPEC、coordinator、JSON Schema、participant、状态机、scope overlap 检测——全部零改动。

### 9. 五场景终极测试（test_ultimate.py）

#### 目标

用一个本地可运行的测试脚本覆盖 5 种不同的多人多 Agent 协作模式，每个场景自包含（独立 server + agents + teardown），有明确的 PASS/FAIL 判定。

#### 五个场景

| # | 场景 | Scope 类型 | 执行模型 | 关键协议特性 | Agents |
|---|------|-----------|---------|-------------|--------|
| 1 | 软件团队代码协作 | file_set | post_commit | CONFLICT_REPORT, STALE_STATE_REF, 自动 rebase | Alice, Bob |
| 2 | 家庭旅游规划 | **task_set** | post_commit | 3-way 冲突检测, coordinator 仲裁, 非文件资源协调 | Dad, Mom, Kid |
| 3 | 多人文档协同编辑 | file_set | post_commit | agent 通用化验证（.md 文件）, 顺序依赖链 | Editor, Reviewer, Formatter |
| 4 | Pre-commit 治理 + 故障恢复 | file_set | **pre_commit** | **OP_PROPOSE** → authorization → OP_COMMIT, agent 崩溃, **INTENT_CLAIM** | Alice, Bob, Charlie |
| 5 | 冲突升级到仲裁者 | file_set | post_commit (governance) | CONFLICT_ACK (disputed), **CONFLICT_ESCALATE**, arbiter **RESOLUTION** | Alice, Bob, Arbiter |

#### 各场景详解

**场景 1（代码协作）**：Alice 和 Bob 同时修改 auth.py，触发 CONFLICT_REPORT。Claude 仲裁后一方 proceed、一方 yield（或双方 proceed 触发 rebase）。验证冲突检测和至少一方提交成功。

**场景 2（旅游规划）**：首次使用 `task_set` scope。3 个 agent 分别规划不同日程（Dad: day 1,2,5; Mom: day 2,3; Kid: day 3,4），day 2 和 day 3 有重叠，触发冲突。coordinator 仲裁后全部 commit。无 Claude 调用，纯协议验证。

**场景 3（文档编辑）**：验证 agent 通用化。3 个 agent 编辑 .md 文件（README.md, design-doc.md）。Editor 先改 README.md，Formatter 在 Editor 版本基础上继续改——验证 state_ref 链式更新在非代码内容上也正常工作。

**场景 4（pre-commit + 故障恢复）**：
1. Server 配置为 `pre_commit` + `governance`
2. Bob 发 OP_PROPOSE → coordinator 返回 authorization → Bob commit（完整 pre-commit 流程）
3. Alice "崩溃"（断开连接 + 回溯 last_seen 触发 liveness 超时）
4. Alice 的 intent 被 coordinator 自动标记为 SUSPENDED
5. Bob 发 INTENT_CLAIM 接管 Alice 的 suspended intent
6. Governance 模式下 Charlie（owner 角色）自动审批 claim
7. Bob 在 claimed scope 上再次 propose + commit

**场景 5（冲突升级）**：
1. Alice 和 Bob 声明重叠 scope → CONFLICT_REPORT
2. 双方 ACK 为 "disputed"（CONFLICT_ACK）
3. Alice 升级到 Arbiter（CONFLICT_ESCALATE）
4. Arbiter 用 Claude 分析双方立场，发布 RESOLUTION
5. 败方 withdraw intent，胜方直接在已有 intent 下 commit

#### 调试过程中发现并修复的问题

1. **task_set scope overlap 检测失败**：`_do_announce_intent` 把 task URI 放进 `Scope.resources` 字段，但 `scope_overlap()` 对 task_set 检查的是 `Scope.task_ids`。修复：根据 scope_kind 分发到正确的字段。

2. **Governance role_policy 格式不匹配**：coordinator 期望 `default_role`（单数），测试传了 `default_roles`（复数），导致所有 agent 的角色降级为 `participant`，INTENT_CLAIM 找不到 owner 角色的审批人。修复：在 open security profile 下不传 role_policy（默认行为：agent 请求什么角色就授予什么角色）。

3. **execute_task() 在 escalation 后创建新冲突**：场景 5 中胜方调用 `execute_task()` 会宣告新 intent，与已有 scope 再次冲突。修复：胜方直接在已有 intent 下用 `_do_commit()` 提交，不走 execute_task 流程。
