### 2. Bug 修复：`_generate_fix` 输出解析（P0）

**问题描述**：

场景 2 中，fangxm 的 agent 在 `task Fix the bug in auth.py` 时，Claude API 返回了带 markdown 解释文字的代码：

```
Looking at the code, I can identify a bug in the `validate_token` function...

```python
"""Authentication module — has several known bugs for testing."""
import hashlib
...
```​
```

旧的解析逻辑只检查 `result.startswith("```")`，不以 ``` 开头就原样使用。结果**整段文本（解释 + 代码块标记 + 代码）被当作 Python 文件内容提交**。

**根因**：prompt 里写了 `"Return ONLY the complete fixed file — no markdown fences, no explanations."`，但 LLM 不总是遵守格式指令。旧代码缺少防御性解析。

**修复**：新增 `_extract_code()` 方法，用正则搜索所有 ` ```python...``` ` 代码块，取最长的一个作为文件内容。如果没找到代码块才 fallback 到旧的首尾 strip 逻辑。

**改动文件**：
- `mpac-package/src/mpac_protocol/agent.py` — 新增 `_extract_code()` 静态方法替代旧的 4 行 strip 逻辑
- `ref-impl/demo/distributed/run_e2e.py` — 同样的修复

**设计原则**：不信任 LLM 一定遵守格式指令，用代码兜底。

### 3. Bug 修复：server 未广播成功的 OP_COMMIT/INTENT_ANNOUNCE（P1）

**问题描述**：

场景 3 测试中发现，Bob 看不到 Alice 的 commit 通知（没有 `>> alice committed changes to: auth.py` 的异步消息）。

**根因分析**：

问题的调用链路：

```
Agent A 发 OP_COMMIT
  → coordinator.process_message() 返回 []（成功，无协议响应）
  → server 路由循环无消息可广播
  → server._apply_file_changes() 广播 FILE_UPDATED（sideband，用 "type" 而非 "message_type"）
  → Agent B 的 _listen() 把它路由到 sideband_inbox
  → _print_event() 从未被调用
  → Bob 看不到通知
```

coordinator 返回 `[]` 是**正确的**——它的职责是处理状态机逻辑（冲突检测、stale ref 校验等），不负责消息转发。问题在 server 层：server 只转发了 coordinator 生成的响应，**漏掉了"把原始入站消息 relay 给其他参与者"这个职责**。

**修复**：server 检测到成功的 `OP_COMMIT`、`INTENT_ANNOUNCE`、`INTENT_WITHDRAW` 后，主动广播原始消息给所有 agent（OP_COMMIT 广播时剥离 `file_changes` 避免传输大文件内容）。

**安全性验证**：
- 发送者自己也会收到广播，但 `_print_event` 里的 `sender == self.principal_id` 过滤会挡住，不会重复打印
- `_print_event` 在 `_listen` 里是先打印再入队（`_print_event(data)` → `protocol_inbox.put(data)`），所以即使消息后来被 `_do_commit` 的 drain 消费掉，通知已经打印过了
- 广播消息进入 `protocol_inbox` 不会干扰正常流程——`_do_commit` 只检查 `PROTOCOL_ERROR` 和 `CONFLICT_REPORT`，其他消息被 drain 时静默丢弃

**改动文件**：
- `mpac-package/src/mpac_protocol/server.py` — 在消息路由逻辑前新增广播判断

**关键认识**：这不是协议问题，也不是 coordinator 问题。协议定义了消息语义，coordinator 正确处理了状态转换，但 server 作为消息路由器没有履行 relay 职责。三层职责清晰：

| 层 | 职责 | 状态 |
|---|------|------|
| 协议 (SPEC) | 定义消息类型和语义 | 无需修改 |
| Coordinator | 状态机逻辑、冲突检测 | 行为正确 |
| Server | 消息路由和转发 | ← 问题在这里，已修复 |

### 5. 讨论：Agent 通用化——从"只能改代码"到"协作做任何事"

在复盘四场景测试时，提出了一个问题：**MPAC 协议只能用来改代码吗？**

答案是**协议本身是内容无关的**（scope 支持 `file_set`、`entity_set`、`task_set`；OP_COMMIT 传的是任意文本内容 + SHA-256 哈希），已有 `run_family_trip.py` demo 证明协议能处理旅游规划这样的非代码场景。

但**当前 pip 包的 agent 实现是写死给 Python 代码用的**。对比两条路径：

| | `run_family_trip.py`（demo） | `mpac-package`（pip 包） |
|---|---|---|
| 场景 | 旅游规划 ✅ | 只能改代码 |
| scope 类型 | `task_set`（`itinerary://day-1`） | `file_set`（`auth.py`） |
| 共享状态 | `ItineraryStore` | `FileStore` |
| 交互方式 | 自动跑完，单机 | 真人交互，跨网络 |

真正绑死在"改代码"上的只有 **agent 层的几段 prompt**，不涉及协议、coordinator 或基础设施：

| 位置 | 现在硬编码的 | 通用化方向 |
|------|-------------|-----------|
| `_generate_fix` system prompt | `"You are given a Python source file. Fix it..."` | 根据 workspace 内容自适应（放 `.py` 就改代码，放 `.md` 就编辑文档） |
| `_generate_fix` user prompt | `"Return the complete fixed Python file."` | `"Return the complete updated file."` |
| `_extract_code` | 只找 ` ```python ` 代码块 | 找任意代码块，或直接返回原文 |
| `_decide_intent` | 假设目标是改文件 bug | 泛化为"分析任务影响哪些资源" |
| `role_description` | `"Security engineer..."` | 用户自定义 |

**核心思路**：把"领域知识"从代码移到 prompt，让 Claude 根据 workspace 内容自动适应。Agent 代码本身不需要知道自己在什么领域——`FileStore` 存的就是"文件名 → 文本内容"，一个 `auth.py` 和一个 `day-1.md` 在存储层没有区别。

**这是一个好信号**：说明协议抽象做对了（SPEC、coordinator、状态机、冲突检测完全不用改），待泛化的只是最上层的 agent prompt。
