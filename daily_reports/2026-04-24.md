`git remote -v` 空。`git fetch origin` 报 `origin does not appear to be a git repository`。

## 根因：`.git/config` 被 OneDrive reject

```bash
$ cat .git/config
cat: .git/config: Permission denied
$ attrib .git/config
A      O     P       .../.git/config         # O = online-only
$ powershell Get-Content .git/config
Get-Content : The cloud sync provider failed to validate the downloaded data.
```

OneDrive 把 `.git/config` 标成 online-only（因为另一台机器上改过），当前机器想 hydrate 但云端 validate 失败。git 本身能读（`git status` 不需要 config），但 `git remote add` 走 mmap 写就爆 `unable to mmap '.git/config': Invalid argument`。

## 恢复

`.git/` 里 OneDrive 给每台旧机器都留了 conflict copy：

```
.git/config                          868 bytes  ← 坏的当前
.git/config-DESKTOP-3H6GDHJ          646 bytes  ← 早版本，可读
.git/config-DESKTOP-3H6GDHJ-2        868 bytes  ← 同样 online-only
```

`cp .git/config-DESKTOP-3H6GDHJ` 覆盖坏的（via PowerShell 绕过 bash 权限），恢复了 remote origin。

5 个 M 文件全是 filemode 从 755 → 644（Git-on-Windows + OneDrive 不保留 exec bit），内容 0 改动。`git config core.filemode false` + `git add --chmod=+x` 解决。

`git pull origin main` → fast-forward 到 `7b571b2`。**总耗时 ~10 分钟**，但后面所有今天的事件都建立在 "git 工作" 之上。

这一段暴露 **2 个独立 bug**：

## Bug A: ExecutionPolicy=Restricted 真的挡 npm + claude

昨天 daily report 写："EP=Restricted 实测能跑 —— `npm.cmd` / `claude.cmd` / `python.exe` / `mpac-mcp-relay.exe` 全是非 `.ps1` binary，EP 不挡。"

**今天现场证伪**：`Get-ExecutionPolicy -List` 所有 scope 都 `Undefined`（默认 Restricted）。`D:\Program Files\nodejs\npm.ps1` 是 .ps1 shim（node 安装器在 Windows 上就这么装的），Restricted 直接拒。昨天"实测能跑"是因为**那台机器上 npm 和 claude 已经装好了**，bootstrap 跑的是 `Get-Command claude`（只查路径不 invoke）这一支 —— 真首装走 `npm install` 才撞。

## Bug B: `$LASTEXITCODE` 看不到 PSSecurityException / CommandNotFoundException

PowerShell 拒执行 `.ps1` 抛 `PSSecurityException`，找不到命令抛 `CommandNotFoundException`。**这两个都是 .NET exception，不是 native 退出**，所以 `$LASTEXITCODE` 保持上一次的值（或 `$null`）。bootstrap 里的：

```powershell
npm install -g @anthropic-ai/claude-code
if ($LASTEXITCODE -ne 0) { Die "npm install failed..." }
```

—— 看到 `$LASTEXITCODE = 0`（从更早的 `& $python --version` 继承），判定成功，继续。然后：

```powershell
claude /login
if ($LASTEXITCODE -ne 0) { Die "claude /login failed..." }
Say "Claude authenticated."
```

`claude /login` 抛 CommandNotFound，`$LASTEXITCODE` 仍是 0，脚本**打印假的 "Claude authenticated"**，最后在 relay 启动时才挂在"not on PATH"。

## 修：2 bug 一起（commit `091db0f`）

### 顶部加 EP=Bypass

```powershell
$ErrorActionPreference = "Continue"

# Windows client SKUs default to EP=Restricted, which silently rejects
# .ps1 scripts. Override to Process scope (evaporates on shell exit).
try {
    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass -Force -ErrorAction Stop
} catch {
    $curr = Get-ExecutionPolicy
    if ($curr -in 'Restricted','AllSigned','Undefined') {
        Write-Host "[mpac] X Group Policy pins EP=$curr; Process override denied." -ForegroundColor Red
        Write-Host "[mpac]   Run once: Set-ExecutionPolicy -Scope CurrentUser RemoteSigned -Force" -ForegroundColor Red
        exit 1
    }
}
```

Process scope 不需要 admin，GroupPolicy override 时降级到 fall-back 错误消息。

### npm install + claude /login 三重判

```powershell
$npmOk = $false
try {
    npm install -g @anthropic-ai/claude-code
    $npmOk = ($? -and ($LASTEXITCODE -eq 0))       # ← $? 捕 throw，$LASTEXITCODE 捕 exit code
} catch {
    Die "npm install failed to invoke: $($_.Exception.Message)"
}
if (-not $npmOk) { Die "npm install failed..." }

# 装完还得从 registry rebuild PATH — User-scope Path 更新只给新 shell
$env:PATH = [Environment]::GetEnvironmentVariable("Path","Machine") + ";" +
            [Environment]::GetEnvironmentVariable("Path","User")

if (-not (Get-Command claude -ErrorAction SilentlyContinue)) {
    Die "npm install exited 0 but 'claude' isn't on PATH..."
}
```

`claude /login` 用同样 pattern。post-install 的 registry-PATH-rebuild 解决"装完了但当前 shell 找不到"的 Windows 特有尴尬。

sync-deploy → Lightsail rebuild → prod 拉 grep `Set-ExecutionPolicy|npmOk|GetEnvironmentVariable` 4/4 全中。

## 修：env var 自动探测（commit `55d32e9`）

bootstrap.ps1 启动 relay 前 4 源探测 git-bash，设 `$env:CLAUDE_CODE_GIT_BASH_PATH`：

```powershell
if (-not $env:CLAUDE_CODE_GIT_BASH_PATH) {
    $bashCandidates = @(
        "$env:ProgramFiles\Git\bin\bash.exe",
        "${env:ProgramFiles(x86)}\Git\bin\bash.exe",
        "$env:LOCALAPPDATA\Programs\Git\bin\bash.exe"
    )
    # 注册表兜底 — Git-for-Windows 装哪都写这个值
    try {
        $gitRoot = (Get-ItemProperty -Path 'HKLM:\SOFTWARE\GitForWindows' -Name InstallPath -EA SilentlyContinue).InstallPath
        if ($gitRoot) { $bashCandidates += (Join-Path $gitRoot 'bin\bash.exe') }
    } catch { }
    # PATH 上的 bash.exe，但滤掉 WSL 的（claude 不认 WSL bash）
    $onPath = (Get-Command bash.exe -EA SilentlyContinue).Source
    if ($onPath -and ($onPath -match 'Git[\\/]bin')) { $bashCandidates += $onPath }

    $bashFound = $bashCandidates | Where-Object { $_ -and (Test-Path $_) } | Select-Object -First 1
    if ($bashFound) {
        $env:CLAUDE_CODE_GIT_BASH_PATH = $bashFound
        Say "git-bash: $bashFound"
    } else {
        Die "git-bash not found — Claude Code CLI on Windows requires it. ..."
    }
}
```

## 绕过：把诊断挪到 bootstrap 里（commit `f58e7f7`）

在 `& mpac-mcp-relay` 启动前加一段 smoke test：

```powershell
function Invoke-ClaudeSmoke {
    $out = ""
    $code = 0
    try {
        $out = ("ping" | & claude -p --dangerously-skip-permissions 2>&1 | Out-String).Trim()
        $code = $LASTEXITCODE
    } catch {
        $out = "INVOCATION FAILED: $($_.Exception.Message)"
        $code = -1
    }
    return @{ Exit = $code; Output = $out }
}

Say "Smoke-testing claude -p (pre-relay sanity check)..."
$smoke = Invoke-ClaudeSmoke
if ($smoke.Exit -ne 0) {
    Write-Host "[mpac] X claude -p smoke test FAILED (exit=$($smoke.Exit))." -ForegroundColor Red
    Write-Host "[mpac]   Output from claude:" -ForegroundColor Red
    ($smoke.Output -split "`n") | ForEach-Object { Write-Host "[mpac]     $_" -ForegroundColor Yellow }
    ...
}
```

Claude CLI 直接在 bootstrap 的终端里跑一次，stdout 被 `2>&1 | Out-String` 抓走并逐行打印。**无论 claude 写到 stdout 还是 stderr，都能看见**。

# 事件四：sessions-dir heuristic 其实是假阳性

昨天 daily report 事件一 Bug 3 的修法：

```powershell
$sessions = Get-ChildItem -Path (Join-Path $claudeDir "sessions") -EA SilentlyContinue
$loggedIn = $claudeInstalled -and $sessions -and $sessions.Count -gt 0
```

当时写："只有真 OAuth 过才写 sessions"。

**今天现场证伪**：KAIYANG 的 `~/.claude/sessions/` 有 **2 个条目**，都是之前测试留下的旧 session。**OAuth token 过期或 revoke 后 session 文件仍在**。bootstrap 判 `$loggedIn = true`，跳过 `claude /login`，relay 启动，每条 chat 消息死在"Not logged in"。

## 修：smoke 匹配到 "not logged in" 就强制跑 /login（commit `de90d3e`）

```powershell
$smoke = Invoke-ClaudeSmoke

if ($smoke.Exit -ne 0 -and $smoke.Output -match "(?i)not logged in|run /login|please log\s*in") {
    Say "Claude reports Not Logged In. Running claude /login now (browser will open)..."
    $loginOk = $false
    try {
        claude /login
        $loginOk = ($? -and ($LASTEXITCODE -eq 0))
    } catch {
        Die "claude /login invocation failed: $($_.Exception.Message)"
    }
    if (-not $loginOk) { Die "claude /login failed or was cancelled." }
    Say "Re-running smoke after login..."
    $smoke = Invoke-ClaudeSmoke        # 重跑
}

if ($smoke.Exit -ne 0) {
    # 第二次还失败才真 Die
    ...
}
```

Smoke 现在是**登录状态的 ground truth**，不再靠 sessions 目录哄自己。sessions heuristic 保留作为 fast path（零条目 → 肯定没登录，提前跳 smoke 直接 /login），但**绝不作为跳过登录的理由**。

KAIYANG 手动 `claude /login` 完 → 再发 chat → **正常回复**。

# 事件五：mpac-mcp 0.2.4 —— 修 relay 的 stderr-only 盲点

事件三的根因（"看不到 claude 真错误"）不在 bootstrap，在 **mpac-mcp 0.2.3 的 relay.py**：

```python
if proc.returncode != 0:
    err = stderr.decode("utf-8", errors="replace").strip()           # ← 只 decode stderr
    log.warning("claude -p exit=%s stderr=%r", proc.returncode, err[:500])
    ...
    return f"[relay] Claude Code failed (exit {proc.returncode}): {err[:400]}"
```

Claude Code CLI 把**好几类**错误写到 stdout（git-bash 缺失、Not logged in、认证过期 …），stderr 是空的。web chat 看到 `[relay] Claude Code failed (exit 1):` 冒号后 null —— **零信息**。

今天 bootstrap smoke test 绕过了症状（错误在 relay 之前就被抓住），但 relay 本体这段如果以后 claude 再写新错到 stdout 还会同样吞。属于定时炸弹，修掉。

## 修（commit `696eea8`）

```python
if proc.returncode != 0:
    err = stderr.decode("utf-8", errors="replace").strip()
    out = stdout.decode("utf-8", errors="replace").strip()
    log.warning("claude -p exit=%s stderr=%r stdout=%r",
                proc.returncode, err[:500], out[:500])
    # 合并两边。stderr 优先（convention），stdout 兜底（常住错误实际位置）。
    if err and out:
        body = f"{err}\n{out}"
    else:
        body = err or out or "(no output on stderr or stdout)"
    hay = (err + " " + out).lower()
    if "login" in hay or "auth" in hay:
        return f"[relay] Claude Code isn't authenticated on this machine...\n\nRaw error: {body[:400]}"
    return f"[relay] Claude Code failed (exit {proc.returncode}): {body[:400]}"
```

## 产品感：诊断层必须在最外围

昨天的 bootstrap.ps1 + mpac-mcp 逻辑已经"正确了 85%"，但今天的真相是：**第一遍就能装对的代码和第二遍才装对的代码，用户体验差十倍**。而决定第一遍成败的，不是"更高级的 bug"，是**边界条件**：

- EP=Restricted 是不是真的不挡 → 假设错
- 旧 sessions 是不是真的代表登录 → 假设错
- git-bash 是不是真的在 C 盘 → 假设错
- claude 错误是不是真的全写 stderr → 假设错

这 4 个假设昨天写代码时都没验证。每次验证成本 < 5 分钟，一起 20 分钟。而今天因为没验证每一个都炸一轮 + 一次部署，总耗时 **~4 小时**。

结论：**凡是与环境变量/文件系统/外部 CLI 交互的地方，假设必须有验证**。不能假设 "应该没问题"。具体到实施：

1. **每个假设点加 runtime probe** —— 今天引入的 smoke test 是典范。不 trust `sessions dir 非空`，直接 `"ping" | claude -p` 问到底。
2. **诊断信息要突出（红色、多行、env 快照）** —— 今天 smoke 错误消息带 `Env: CLAUDE_CODE_GIT_BASH_PATH=...`，是"失败时一眼看见真相"的关键。默认的 relay error "exit 1 stderr=''" 是反面教材。
3. **日志路径不能只有一条** —— relay 0.2.3 只 decode stderr = 单路径失败即全盲。0.2.4 合并 stderr+stdout = 冗余覆盖。

今天的另一层产品感：**假设错了不丢人，不加 probe 是真的要命**。4 小时里我完全可以接受"又撞到新 Windows 坑"，完全不能接受"撞到新坑但完全看不见"。把诊断层做厚是今天最值钱的投资。
