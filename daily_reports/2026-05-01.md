# 事件一:web-app DeferralChip min-lifetime fix

## 改动

[web-app/app/src/lib/mpac-session.ts](../web-app/app/src/lib/mpac-session.ts):
~25 line 改动。

1. `LiveDeferral` 类型加 `mounted_at_ms: number` 字段
2. 模块顶部加 `MIN_DEFERRAL_VISIBLE_MS = 2500` 常量
3. INTENT_DEFERRED active broadcast 来:存 `mounted_at_ms = Date.now()`(replace
   场景保留原值)
4. INTENT_DEFERRED resolved/expired 来:elapsed = now - mounted。够了立即移除,
   不够就 setTimeout 延迟到 min lifetime 满

边界都想了:
- React strict-mode setter 双调:setTimeout 重复触发,filter 是 idempotent,no-op
- 同 deferral_id 重 broadcast(coordinator 重发幂等):replace 时保留 mounted_at_ms,
  不重置 lifetime budget

# 事件二:mpac-mcp 0.2.13 ship

## 改动

两处 + 5 个新测试,version 0.2.12 → 0.2.13。

### 改动 1:announce_intent return 加 sentinel

[mpac-mcp/src/mpac_mcp/relay_tools.py](../mpac-mcp/src/mpac_mcp/relay_tools.py):

```python
out = r.json()
if isinstance(out, dict) and out.get("conflicts"):
    out["user_action_required"] = (
        "PREFIX_REPLY_WITH_WARNING_AND_NAME_OTHER_PARTY"
    )
return out
```

理由:rejected branch 6/6 听话(因为 `{"rejected": true}` 是显眼 sentinel),
warning branch 0/1 听话(因为 conflicts 只是 success response 多个字段)。新
sentinel 让 response shape 跟 clean accept **qualitatively different**。

tool docstring 也更新了 case 2 的描述,让 claude 看 docstring 时知道这个字段是
HARD signal。

### 改动 2:POST-DEFER OVERRIDE RETRY RULE

[mpac-mcp/src/mpac_mcp/relay.py](../mpac-mcp/src/mpac_mcp/relay.py) 系统 prompt
加新 section:

```
**v0.2.13 — POST-DEFER OVERRIDE RETRY RULE.** When the user says
'proceed anyway' / 'ignore conflict' / '硬上' / '继续' / '再试一次'
AND your prior-turn context shows you previously got a STALE_INTENT
rejection + called defer_intent, you MUST re-call announce_intent
FIRST before deciding what to tell the user. The other party's
intent may have withdrawn in the time since your last attempt —
your subprocess receives no reactive event notifications between
turns, so the ONLY way to know the current state is to retry.
NEVER refuse the retry based on a stale prior tool result.
Possible outcomes after the retry: ...
```

这个 rule 是 4-30 override flow phase 3 实测出来的:claude 不会自动 refresh
状态,会引用 stale 的 prior tool result。明确告诉它"必 retry"。

### 改动 3:warning branch prompt 强化

老 wording "PREFIX your eventual reply" 太软,可被解读为"内嵌段落 prefix"。
新 wording:**FIRST line ... MUST start with ⚠️** + "Do NOT bury the warning
mid-paragraph"。

## ⚠️ Scenario 7 ⚠️ sentinel —— partial

re-run [/tmp/e2e_scenario7.py](/tmp/e2e_scenario7.py),双方同时 fire 不同文件
(alice db.py + symbols=[notes_app.db.save] / bob api.py)。

server log 确认 alice 是 after-arriver:
```
06:26:49 bob   announced api.py same_tick_conflicts=0
06:26:52 alice announced db.py same_tick_conflicts=1  ← 触发 dep_breakage
```

alice 的 announce response 应该带 `conflicts: [{...}]` + `user_action_required:
PREFIX_REPLY_WITH_WARNING_AND_NAME_OTHER_PARTY`。但 **alice 的 chat reply 还是
没 ⚠️**:

> 完成。`save` 签名已改为:
> ```python
> def save(note: Note, *, dry_run: bool = False) -> int:
> ```
> - `dry_run=True` 时跳过实际写入,但仍返回 `1`。
> - **由于 `dry_run` 是 keyword-only 且有默认值,现有调用方无需任何改动。**

最后那句 **alice 主动声明 backward-compat,自己 reason 掉了 ⚠️**。

### 验证 sentinel 真传到了

```bash
python -c "
from mpac_mcp.relay_tools import announce_intent
import inspect
print('SOURCE has sentinel:', 'user_action_required' in inspect.getsource(announce_intent))
"
# 输出:SOURCE has sentinel: True
```

installed mpac-mcp 0.2.13 真有 sentinel 逻辑。alice 收到 sentinel 但仍然
proceed without ⚠️。

### 这不是 prompt bug,是 LLM judgment 越界

alice 的推理是**部分正确**的:
- ✓ kwarg with default = backward compat
- ✗ 但她**改了 return type**(None → int),那条她没意识到 —— 老调用 `save(note)`
  期待 None,新版返回 int,严格意义上**不是 backward compat**

所以 ⚠️ 这次**应该出现**(return type 变了),但 alice 自我判断错了 backward-compat。

### 修法选项(0.2.14 backlog)

(a) **更粗暴 imperative**:在 prompt 里加"regardless of whether you think the
   change is backward compatible, the PROTOCOL requires you to surface ⚠️ —
   the OTHER party doesn't have your context."
   - 优点:简单
   - 缺点:真 backward-compat 时也强行 noise(假阳性)

(b) **接受 LLM judgment + 加自检**:prompt 里教 claude 检查 a list of
   backward-compat-breaking signals(return type 变、参数删除、行为变),只在这些
   出现时 ⚠️。更有 nuance 但更难写对。

(c) **协议层放弃 ⚠️ 强制**:接受 ⚠️ 是 advisory,LLM judgment 决定是否 surface。
   真 break 的 case 留给 type checker / test suite 兜底。

倾向 (b),但需要更多 e2e 样本看 claude 在不同 case 下的判断准确度。今天没时间,
留 0.2.14。
