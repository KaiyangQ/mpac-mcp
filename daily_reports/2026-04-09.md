### 4. 发现并修复 intent 生命周期 UX 问题

**问题描述：**

场景 2 测试结束后，合作者停止操作。Alice（host 端）重新输入同一个 task（目标仍是 `auth.py`），**再次触发了 CONFLICT_REPORT**，但此时没有任何其他人在下任务。这很反直觉——明明对方已经"做完了"，为什么还在冲突？

**根因分析：**

- MPAC 协议中，`INTENT_ANNOUNCE` 的语义是 **"我计划在这个范围工作"**，不是 "我执行一次操作"
- `OP_COMMIT` 只结束 operation 的生命周期，**不会自动结束父 intent**——协议允许一个 intent 下有多个 operation
- 所以第一轮测试后，合作者的 intent 虽然没再下新任务，但在 coordinator 状态机里依然是 `ACTIVE`，**继续占着 `auth.py` 这个 scope**
- Alice 再宣告新 intent 时，与合作者的"幽灵 intent"发生 scope 重叠 → 触发冲突
- 这在协议层面是**完全正确的行为**——coordinator 忠实执行规则

**修复（agent 层，不改协议）：**

在 `mpac-package/src/mpac_protocol/agent.py` 的 `_run_task_interactive` 末尾加上：

```python
# Withdraw the intent — task is done, scope should no longer be "claimed".
try:
    msg = self.participant.withdraw_intent(
        self.session_id, intent["intent_id"], "task_completed"
    )
    await self._send(msg)
except Exception as e:
    self.log.debug(f"withdraw_intent failed: {e}")

# Clear stale conflict reports from this task
self.conflicts_received = []
```

同时在任务开始时（`_run_task_interactive` 开头）也清空 `conflicts_received`，避免上一轮 stale 冲突污染。

### 5. 关键洞察：协议本身已经完整覆盖了这个场景

在讨论修复方案时，收到合作者一个很好的提问：**"既然一个 intent 下可以有多个 operation，能不能在所有 operation 结束后自动结束 intent？"**

这引发了一轮有价值的讨论，最终得到一个重要认识：**MPAC 协议早就有 `INTENT_WITHDRAW` 消息**，完全支持"任务结束时主动结束 intent"的语义。问题**不在协议**，而在我们的 agent 实现——`_run_task_interactive` 任务完成后忘了发 `WITHDRAW`。

可以用一个直观的类比来理解：

```
聊天场景                         MPAC 场景
──────────────────              ───────────────────
你提问                           用户下达 task
Claude 思考 + 输出               Agent 宣告 intent
Claude 输出内容                   Agent 提交 operation
Claude 输出 <end_turn>  ←───→    Agent 发送 INTENT_WITHDRAW
你看到回答结束                    Coordinator 看到 intent 结束
```

**`INTENT_WITHDRAW` 在 MPAC 里扮演的角色，等同于 Claude API 里的 `<end_turn>` token**——由生产者（agent）主动发出的"我说完了"信号。协议设计者早就想到了这个需求，只是我们写交互式 CLI 的时候没把这个信号接上。

**关于"所有 operation 完成就自动结束 intent"的最终结论：不需要改协议**

1. **coordinator 无法可靠推断"agent 是否还想做下一个 op"**——operation 之间的思考间隙和"真正做完"在 coordinator 视角下无法区分
2. **显式信号比推断更干净**——这是分布式系统的通用设计原则（Git branch、数据库事务、HTTP session 都是显式结束，不靠"最后活动"推断）
3. **协议已经提供了机制**——`INTENT_WITHDRAW` 是显式信号，`ttl_sec` 是超时兜底，组合起来足够覆盖所有场景
4. **强加"auto-close on all-ops-terminal"会损害灵活性**——会破坏"一个 intent 多个 op"的合法用法，比如长期"重构 auth 模块"的 intent 下分多次提交

这个讨论的价值在于：**明确了协议和实现的边界**——当某个需求看起来像协议缺陷时，先检查协议是否已经提供了机制但实现没用上。这次就是典型案例。

### 6. 为什么本地测试没暴露这个问题？（方法论教训）

**本地自动化测试（`test_e2e.py`）** 每次是"一次性跑完就退出"：
- 两个 agent 同时启动
- 同时执行任务
- 同时退出（WebSocket 连接关闭）
- coordinator 清理状态，老 intent 自然消失

这种"run-to-completion"的测试模式**看不出 intent 在长连接内的堆积问题**。

**跨网络真实联调场景完全不同：**
- 两个人保持长连接
- 连续下多个 task（每次任务之间不重启 agent）
- 没有"退出-重启"的自然清理时机
- 老 intent 就堆积在 coordinator 状态里，形成"幽灵 intent"干扰新任务

**教训：**
- 以后设计测试时，应该有一类"长连接 + 多任务序列"的测试，模拟真实使用模式
- 自动化测试和真实使用的 gap，往往藏在**连接生命周期**和**会话内的状态累积**上
- 这类 UX bug 很难在 CI 里发现，只有真实场景才能暴露

### 8. End-to-end 使用流程 walkthrough（dev 自用参考）

**为什么记在这里**：晚些时候梳理"如果外部用户真的 `pip install` 了这个包，两个不同局域网的人具体怎么协作"这个问题，把完整交互路径从代码里读了一遍（`mpac-package/src/mpac_protocol/agent.py` 和 `server.py`）。这份 walkthrough 不是给外部用户看的 how-to——对外的简短版本留在 `examples/two_machine_demo/README.md`——而是 dev 自己的参考，包括了那些"外部用户会碰到但 README 里不会写"的 limitation 清单。与第 4 节那个 intent 生命周期 bug 的发现属于同一类"自动化测试覆盖不到的现实使用路径"问题，所以归档在同一份 daily report 里。

#### 心智模型：client-server，不是 P2P

MPAC **不是** P2P 协议，是 client-server 模型。两个人必须一个当 host，另一个当 guest：

```
┌────────────────────────────────┐          ┌──────────────────────────┐
│          HOST machine          │          │       GUEST machine      │
│  (Alice @ 家里 WiFi)            │          │ (Bob @ 咖啡厅 WiFi)        │
│                                │          │                          │
│  ┌──────────────────────────┐  │          │  ┌────────────────────┐  │
│  │   MPACServer (coordinator)│  │          │  │   MPACAgent (Bob)  │  │
│  │   + FileStore (内存)      │◄─┼── WS ────┼──┤                    │  │
│  │   + Alice's MPACAgent    │  │          │  └────────────────────┘  │
│  └──────────────────────────┘  │          │                          │
│  有 pip 包 + Anthropic key      │          │  有 pip 包 + Anthropic key │
│  决定"哪个目录是共享的"         │          │  没有本地 workspace 副本  │
└────────────────────────────────┘          └──────────────────────────┘
                │                                       ▲
                │   ngrok / cloudflared tunnel          │
                └───── wss://xxxx.ngrok-free.dev ──────┘
```

几个关键 fact（外部用户第一次用时最容易踩错的认知）：

1. **只有 host 的目录是"共享的"**。Host 启动时 `--workspace /path` 指定一个目录，整个目录被 load 进内存（`FileStore.load_directory`, `server.py:31-85`）。Guest 没有这个目录的本地副本，所有文件内容都是 runtime 从 host 抓过来的（sideband `FILE_LIST` / `FILE_READ`, `server.py:122-137`）。
2. **所有状态都在 host 内存里**。Host 进程关闭 = session 状态蒸发。clean quit 时 host 会把最终状态写到 `./output/` 和 `./output/transcript.json`（`test_site_A/run.py:91-94`），crash 就啥都没了。
3. **两个人的 agent 各用各的 Anthropic API Key**。没有 key 共享，每个人用自己的 Claude 额度付费。
4. **`mpac-starter-kit.zip` 虽然叫 kit 但它装的是 guest 端**（只有 `MPACAgent`）——内部其实是同一个 pip wheel，区别只是附带了一个 guest 端 `run.py`。

#### 交互阶段：view / task / quit

Interactive CLI **只有 3 个命令**（`agent.py:438-441`）：

| 命令 | 作用 |
|---|---|
| `view <filename>` | 打印某个文件的当前内容（从 host 抓） |
| `task <description>` | 扔一个自然语言任务给自己的本地 agent |
| `quit` | 离开 session |

没有 `status` / `history` / `list` / `diff` / `cancel`——**current CLI 就这三个**。要加功能都得改 `_run_task_interactive` 或者加新的 shell 命令分支。

**`task` 命令会触发 4 个 phase**（`agent.py:485-609`）：

1. **[1/4] Analyzing** —— 调 Claude `_decide_intent` 决定要改哪些文件，生成 plan
2. **[2/4] Reading** —— 从 host 抓当前要动的所有文件，打印出来给用户看
3. **[3/4] Announcing intent** —— 发 `INTENT_ANNOUNCE` 给 coordinator，然后**等 5 秒**让其他 agent 响应（`_drain_conflicts(5.0)`）
4. **[4/4] Generating fixes** —— 对每个目标文件调 Claude `_generate_fix`，显示彩色 diff，commit via `OP_COMMIT`；如果收到 `STALE_STATE_REF` 就自动 rebase 最多 2 次（`agent.py:553`）；task 结束发 `INTENT_WITHDRAW`（第 4 节的修复点）

#### 场景 A：无冲突协作

Alice：
```
[Alice] > task Fix the token expiry bug in auth.py
[Alice 看到 4 个 phase，最后 Committed! auth.py -> sha256:abc...]
```

**同时 Bob 的 terminal 异步冒出**（`_print_event`, `agent.py:148-176`）：
```
  >> alice announced intent: Fix token expiry validation
     Files: ['auth.py']
  >> alice committed changes to: auth.py (sha256:abc...)
```

这时 Bob 可以 `view auth.py` 看新版本，或者开始自己的任务。

#### 场景 B：两人 task 撞了

Alice 和 Bob 同时：
```
[Alice] > task Refactor auth.py to use JWT
[Bob]   > task Fix the bug in auth.py
```

两人都走到 `[3/4] Announcing intent...`，coordinator 检测 scope 重叠并 broadcast `CONFLICT_REPORT`。两边都会看到：
```
  CONFLICT with another agent!
    Category: scope_overlap
    Between: intent-alice-xyz vs intent-bob-abc
```

然后**各自的本地 agent 独立调用 Claude 做仲裁**（`_decide_conflict`, `agent.py:310-323`），返回 `"yield"` 或 `"proceed"`。如果一方 yield，那方的 task 取消并发 `INTENT_WITHDRAW`。如果两边都 proceed，按 first-committer-wins 规则，晚的那个收到 `STALE_STATE_REF` 自动 rebase（最多 2 次）。

**整个冲突→仲裁→rebase 流程在 log 里是完全可见的，没有 silent override**——这是 MPAC 相对"两个 agent 各改各的然后 git merge"的核心价值。

#### 场景 C：Host 退出

```
[Alice] > quit
  Alice leaving session...
  [Bob 那边立刻看到 >> alice left the session]
```

Alice 的 `run.py` 在 agent 结束后会：
1. 把 `FileStore` 里所有文件的当前状态写到 `./output/` 目录（`server.file_store.save_to_directory`）
2. 把完整 transcript（每条 MPAC envelope）写到 `./output/transcript.json`
3. 关 WebSocket server

Bob 那边 WebSocket 断开，`websockets.exceptions.ConnectionClosed`，`run.py` 结束。

**重点**：Alice 原来的 `~/my_project` 文件夹**没有自动写回**（`save_to_directory` 写的是 `./output/` 不是 `workspace_dir`）。要最终成果得手动从 `./output/` 拿。这对"我和同事 pair programming 实时改代码"的预期是不 match 的——见下面 limitation #1。

#### 给外部用户不会在 README 里写但他们必定踩的 8 个坑

1. **Workspace 是单向的**。Host load 一次就进内存，之后变更全在内存，**原目录文件没被同步写回**。要最终成果得手动从 `output/` 拿。外部用户如果把 `~/my_project` 当生产目录分享，会以为 agent 的修改直接落盘，实际不会。
2. **没有 reconnection**。ngrok 断了或 Bob WiFi 掉了，没有自动重连。要重启 guest 脚本，用之前的 session_id 再连一次。
3. **Host 崩了 session 就没了**。`FileStore` 只在内存里。Crash = 状态蒸发。`ref-impl/python/mpac/coordinator.py` 有完整的 snapshot + audit log replay，但 pip 包里的 `MPACServer` 是简化版，**没接上 snapshot**。只在 clean `quit` 时才保存。
4. **CLI 就 3 个命令**。`view` / `task` / `quit`，没了。Demo-quality 的 shell，不是 daily driver。
5. **每次 task 至少烧 3 次 Claude 调用**：`_decide_intent` + `_generate_fix` + 如果有冲突再加 `_decide_conflict`。每次 rebase 再加一次 `_generate_fix`。账单取决于 task 频率和文件大小。
6. **Session ID 明文商量**。默认 `collab-session-001`，两个不同的 host 同时跑同一 session id + 猜到 ngrok URL 就可能连错。建议 host 用 `--session-id $(uuidgen)`。
7. **ngrok free tier URL 2 小时过期**。长期协作要升级 ngrok、用 cloudflared tunnel（永久 URL）、或 VPS。
8. **Guest 完全依赖 host 在线**。Host 关电脑 = session 没了，guest 独立做不了任何事。不是 P2P，是集中式。

**总结**：这个 walkthrough 让我看清楚了 demo-quality 和 production-quality 之间的 gap 在哪里——主要是 #1（workspace 不写回）、#3（crash = 状态蒸发）、#4（CLI 功能太少）。这三个是 blocker-level 的，修之前这个包只能当 demo / 研究工具，不能当生产协作工具。对外宣传要写清楚"draft / experimental"（我们 SPEC.md 和 pyproject.toml classifier 都写了 Alpha，所以定位是诚实的）。
