### 4. 发现并修复 intent 生命周期 UX 问题

**问题描述：**

场景 2 测试结束后，合作者停止操作。Alice（host 端）重新输入同一个 task（目标仍是 `auth.py`），**再次触发了 CONFLICT_REPORT**，但此时没有任何其他人在下任务。这很反直觉——明明对方已经"做完了"，为什么还在冲突？

**根因分析：**

- MPAC 协议中，`INTENT_ANNOUNCE` 的语义是 **"我计划在这个范围工作"**，不是 "我执行一次操作"
- `OP_COMMIT` 只结束 operation 的生命周期，**不会自动结束父 intent**——协议允许一个 intent 下有多个 operation
- 所以第一轮测试后，合作者的 intent 虽然没再下新任务，但在 coordinator 状态机里依然是 `ACTIVE`，**继续占着 `auth.py` 这个 scope**
- Alice 再宣告新 intent 时，与合作者的"幽灵 intent"发生 scope 重叠 → 触发冲突
- 这在协议层面是**完全正确的行为**——coordinator 忠实执行规则

**修复（agent 层，不改协议）：**

在 `mpac-package/src/mpac_protocol/agent.py` 的 `_run_task_interactive` 末尾加上：

```python
# Withdraw the intent — task is done, scope should no longer be "claimed".
try:
    msg = self.participant.withdraw_intent(
        self.session_id, intent["intent_id"], "task_completed"
    )
    await self._send(msg)
except Exception as e:
    self.log.debug(f"withdraw_intent failed: {e}")

# Clear stale conflict reports from this task
self.conflicts_received = []
```

同时在任务开始时（`_run_task_interactive` 开头）也清空 `conflicts_received`，避免上一轮 stale 冲突污染。

### 5. 关键洞察：协议本身已经完整覆盖了这个场景

在讨论修复方案时，收到合作者一个很好的提问：**"既然一个 intent 下可以有多个 operation，能不能在所有 operation 结束后自动结束 intent？"**

这引发了一轮有价值的讨论，最终得到一个重要认识：**MPAC 协议早就有 `INTENT_WITHDRAW` 消息**，完全支持"任务结束时主动结束 intent"的语义。问题**不在协议**，而在我们的 agent 实现——`_run_task_interactive` 任务完成后忘了发 `WITHDRAW`。

可以用一个直观的类比来理解：

```
聊天场景                         MPAC 场景
──────────────────              ───────────────────
你提问                           用户下达 task
Claude 思考 + 输出               Agent 宣告 intent
Claude 输出内容                   Agent 提交 operation
Claude 输出 <end_turn>  ←───→    Agent 发送 INTENT_WITHDRAW
你看到回答结束                    Coordinator 看到 intent 结束
```

**`INTENT_WITHDRAW` 在 MPAC 里扮演的角色，等同于 Claude API 里的 `<end_turn>` token**——由生产者（agent）主动发出的"我说完了"信号。协议设计者早就想到了这个需求，只是我们写交互式 CLI 的时候没把这个信号接上。

**关于"所有 operation 完成就自动结束 intent"的最终结论：不需要改协议**

1. **coordinator 无法可靠推断"agent 是否还想做下一个 op"**——operation 之间的思考间隙和"真正做完"在 coordinator 视角下无法区分
2. **显式信号比推断更干净**——这是分布式系统的通用设计原则（Git branch、数据库事务、HTTP session 都是显式结束，不靠"最后活动"推断）
3. **协议已经提供了机制**——`INTENT_WITHDRAW` 是显式信号，`ttl_sec` 是超时兜底，组合起来足够覆盖所有场景
4. **强加"auto-close on all-ops-terminal"会损害灵活性**——会破坏"一个 intent 多个 op"的合法用法，比如长期"重构 auth 模块"的 intent 下分多次提交

这个讨论的价值在于：**明确了协议和实现的边界**——当某个需求看起来像协议缺陷时，先检查协议是否已经提供了机制但实现没用上。这次就是典型案例。

### 6. 为什么本地测试没暴露这个问题？（方法论教训）

**本地自动化测试（`test_e2e.py`）** 每次是"一次性跑完就退出"：
- 两个 agent 同时启动
- 同时执行任务
- 同时退出（WebSocket 连接关闭）
- coordinator 清理状态，老 intent 自然消失

这种"run-to-completion"的测试模式**看不出 intent 在长连接内的堆积问题**。

**跨网络真实联调场景完全不同：**
- 两个人保持长连接
- 连续下多个 task（每次任务之间不重启 agent）
- 没有"退出-重启"的自然清理时机
- 老 intent 就堆积在 coordinator 状态里，形成"幽灵 intent"干扰新任务

**教训：**
- 以后设计测试时，应该有一类"长连接 + 多任务序列"的测试，模拟真实使用模式
- 自动化测试和真实使用的 gap，往往藏在**连接生命周期**和**会话内的状态累积**上
- 这类 UX bug 很难在 CI 里发现，只有真实场景才能暴露
