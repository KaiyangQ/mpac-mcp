# Daily Report — 2026-04-07

## 概述

今天在 MPAC 协议中新增了 **Backend Health Monitoring** 功能（v0.1.13），集成 aistatus.cc 的开源 AI 状态监控 API，使 agent 能够声明后端 AI 模型依赖、汇报后端健康状态，并通过 coordinator 进行告警和模型切换治理。

### 1. Backend Health Monitoring 功能设计与实现

**新增协议扩展（全部向后兼容）：**

- **HELLO 扩展**：新增可选 `backend` 字段（`model_id`、`provider`），agent 加入 session 时声明 AI 模型依赖
- **HEARTBEAT 扩展**：新增可选 `backend_health` 字段（`model_id`、`provider_status`、`status_detail`、`checked_at`、`alternatives`、`switched_from`、`switch_reason`），定期汇报后端状态
- **COORDINATOR_STATUS 扩展**：`event` 枚举新增 `backend_alert`，附带 `affected_principal` 和 `backend_detail` 条件字段
- **Liveness Policy 扩展**：`backend_health_policy` 对象（`enabled`、`check_source`、`check_interval_sec`、`on_degraded`、`on_down`、`auto_switch`、`allowed_providers`）
- **新增错误码**：`BACKEND_SWITCH_DENIED`（auto_switch=forbidden 或 provider 不在白名单）
- **新增 SPEC Section 14.3.1**：完整的行为规范，包括 coordinator 动作（ignore/warn/suspend_and_claim）和模型切换治理（allowed/notify_first/forbidden）

**协议 vs 实现的职责边界：**
- 协议管：信号机制（声明、汇报、告警）+ 治理规则（白名单、切换策略）
- 实现管：选哪个替代模型、什么时候触发切换、是否切回

**`provider_status` 枚举和 `alternatives` 结构直接对齐 aistatus.cc `/api/check` 响应格式。**

### 6. Coordination Overhead vs Decision Time Demo

**新增 demo：** `ref-impl/demo/distributed/run_overhead_comparison.py`

为 paper 的核心 claim 提供实证支撑：**MPAC 消除 coordination overhead，但不压缩 decision time。**

**设计：**
- 同一场景（3 个 AI agent 审批跨模块重构 PR）跑两遍
- Mode A（Traditional）：串行执行，agent 等待前序完成，来回澄清冲突，事后发现矛盾返工
- Mode B（MPAC）：通过 WebSocket coordinator 并行执行，INTENT_ANNOUNCE 阶段预检冲突，structured conflict position 并行提交
- 每次 Claude API 调用精确计时，标记为 `decision_time`；所有等待/传递/重建标记为 `coordination_overhead`
- 两种 mode 使用完全相同的 Claude prompt，保证 decision quality 可比

**代表性结果（Claude Sonnet, 3 agents × 3 phases）：**

| 指标 | Traditional | MPAC | Delta |
|------|-----------|------|-------|
| Decision Time | 60.2s | 54.9s | -9%（API 噪声） |
| Coordination Overhead | 65.1s | 3.0s | **-95%** |
| Wall Clock | 125.3s | 25.7s | -79% |
| OH / Wall Clock | 52.0% | 11.7% | |

结论：decision time 两边统计等价，wall clock 的改善完全来自 coordination overhead 的消除。
