# 事件零:Windows bootstrap `claude /login` TUI 不退

> 早上一开门,user 报告 Windows 测试者在跑 `iex (irm '...bootstrap.ps1')` 时,

## 因果链分析

读 [web-app/api/routes/agent.py:1026-1050](web-app/api/routes/agent.py:1026)
的 `Invoke-ClaudeLogin`:

```powershell
$proc = Start-Process -FilePath "cmd.exe" `
    -ArgumentList "/c","claude /login" `
    -Wait -NoNewWindow -PassThru -ErrorAction Stop
```

`Start-Process -Wait` 等子进程退出。但 **`claude /login` 是 TUI(alternate
screen)**,OAuth 浏览器流程完成后**不自动退出**,把用户留在 Claude REPL 主界面。
用户必须在 TUI 里手动 `/exit` 或 Ctrl+C → 子进程退出 → `Start-Process -Wait`
返回 → bootstrap 继续。

4-25 那次 `Start-Process` 修复(see daily report 4-25)解决的是 TUI stdio entangle
问题(TUI 显示乱码)。修完 TUI 显示正常了,**但 "OAuth 完不自退" 这个第二层问题
没人处理** —— bootstrap 把这个职责完全丢给 `claude /login` 自己。

## 三个修法

| 方案 | 工作量 | 取舍 |
|---|---|---|
| **A. 加用户提示** | 1 行 `Say` | 最便宜,但用户得看终端 + 手动操作 |
| **B. Watchdog 监 token 文件** | ~30 行:并行 PS Job 轮询 `~/.claude/sessions/`,出现就 `Stop-Process` | 自动化;但硬 kill 可能在 fsync 还没完成时打断 token,时序不稳 |
| **C. 切到非交互式 login 命令** | 取决于 Claude Code CLI 是否提供 | 最干净,如果存在 |

我用 `claude-code-guide` agent 查 C —— `claude --help`、`claude auth --help`、
`claude /login --help` 都没找到 `--no-tui` / `--exit-after-auth`。但找到
**`claude setup-token`** 是 Anthropic 官方为 CI/headless 设计的命令,**OAuth 完成
后立即 exit**。

但 follow-up 验证发现:
- `claude /login` → 写 `~/.claude/.credentials.json`,后续 `claude -p` 直接用
- `claude setup-token` → **只 print token 到 stdout,不写 credentials.json**;
  必须设 `CLAUDE_CODE_OAUTH_TOKEN` env var 后续才认证

**两者不是 drop-in replaceable**。换 setup-token 要做 secret capture + 持久 env

# 事件四:用例 4 — INTENT_DEFERRED yield chip(3 轮迭代!)

> 重点测的用例,直接暴露了 v0.2.9 system prompt 的一个 LLM 行为 bug。

## Round 1 — Bob 嘴上让路但 **没调** `defer_intent` ❌

测试 prompt 让 Alice 干个 20-30 秒的 task(改 2 个函数 + 强制 read 文件),Bob
等 5 秒后用 prompt:**"如果别人在改这个文件,直接放弃,什么都不要做"**。

期望:Bob 的 Claude check_overlap → 看到 Alice → 调 `defer_intent` → yield chip 出现。

实际(20:53-20:54 jsonl):
```
20:53:53.953  Alice's Claude → INTENT_ANNOUNCE on db.py
20:53:55.140  Bob 发 chat (Alice announce 完 1.2 秒后,时机正确)
20:54:13.380  Alice's Claude → INTENT_WITHDRAW reason=done

Bob 这一侧:🔇 完全没出现在协议层
  - 没有 INTENT_ANNOUNCE (✓ 符合"不 announce"预期)
  - 没有 INTENT_DEFERRED (❌ 期望它调 defer_intent)
```

Bob 的 Claude 嘴上说"等他完成",**但没调任何 MPAC 工具**。

**根因**:
- mpac-mcp 0.2.9 system prompt 第 2 步说:`If overlap exists AND you decide to YIELD: call defer_intent`
- 但用户 prompt 写:**"直接放弃,什么都不要做"**
- Claude 把"什么都不要做"字面理解 = 连 `defer_intent` 也不调

**这是 prompt 优先级冲突**:user prompt 比 system prompt 优先级高,Claude 跟着
user prompt 走。

## Round 2 — 改强 user prompt,验证协议路径完整 ✓

user prompt 改成显式列步骤:**"先 check_overlap → 如果有 overlap,调 `defer_intent`
+ 回我'已让路'+ 不要 announce/不要改文件"**。

实际(21:08 jsonl):
```
21:08:46.357  Alice announce         intent-agent-1-4bd69ab614  on db.py
21:08:54.683  Bob   DEFER_INTENT     defer-agent-2-a70349e56d   ← 8s 后,正确调用!
              observed_intent_ids: ['intent-agent-1-4bd69ab614']  ← 精确捕获 Alice intent_id
              reason: "Alice 正在改 db.py，让路"
21:08:54.683  ↑ Coordinator broadcast (status=active, expires_at = +60s TTL)
21:08:58.961  Alice withdraw         intent-agent-1-4bd69ab614  reason=done (4.3s 后)
21:08:58.961  ↑ Coordinator broadcast (status=RESOLVED, reason="observed_intents_terminated")
              ← 这一条让 yield chip 立即消失,不用等 60s TTL ⭐
```

**意外惊喜**:server 在 Alice withdraw 时**自动 emit RESOLVED**,yield chip 不
等 60s TTL 直接消失。这是 4-28 daily report 续事件 D 的「待办」清单第 2 条:

> defer 晚于观察 intent withdraw 时,server 立刻 resolve —— 当前靠客户端 TTL 兜底
> (60s 才清)。可以加一条:`_handle_intent_deferred` 里检查 `observed_intent_ids`
> 是不是已经全部 terminal,是的话立刻 emit `status="resolved"`。10 行小改进,留待
> 下次 cut 0.2.7-mpac 时顺手做。

但今天测出来 server 不只在「defer 晚于 withdraw」case 工作,**正常顺序(defer 先,
intent 后 withdraw)的 case 也立即 emit RESOLVED** —— 因为
`_cleanup_deferrals_for_terminated_intent` 挂在 `_handle_intent_withdraw` 上,对所有
deferral 扫 observed_intent_ids,命中就立刻广播 resolved。User 浏览器侧确认了 yield
chip 灰色虚线 + Alice 完成时立即消失。

## User 一个深问题:**"既然有 dependency_breakage,两边都改了,为什么不冲突?"**

这个问题触及 MPAC 协议的核心哲学,值得展开讨论:

**直接答案**:这次实际不 break,因为 Alice 的改动是 backward-compatible:
- 加了 `*, dry_run=False` keyword-only 参数(有默认值)→ Bob 的 `db.save(note)`
  fallback 默认值 ✓
- 返回类型 `None → int` → Bob 不用返回值 ✓

**协议事先不知道这件事**,只能基于「Alice 在改 `notes_app.db.save`,Bob 在调
`notes_app.db.save`」做保守判断。**宁可误报,也不漏报**。

**MPAC 是软约束(advisory protocol),不是数据库锁**。故意如此设计:

| 如果是硬阻止 | 实际做软约束 |
|---|---|
| Alice 改 `save()` 时阻止 Bob 调 `save()` | 警告 Bob,Bob 可以 yield 或 proceed |
| 容易死锁(A 等 B,B 等 A)| 决策权在用户/agent 手里,不死锁 |
| Agent 自主性受限 | Agent 自由,但留 audit trail |
| 假阳性也阻止 = 卡住开发流 | 假阳性只是一张红卡,Acknowledge 一下就过 |

**协议的工作是降低协调成本,不是消除冲突**。真正的冲突解决靠人 + agent 的语义
判断,不是字节级 lock。

如果 Alice 的改是真 break(把 `save()` 删了改成 `persist()`),Bob 的代码会
runtime AttributeError —— 这就是「软约束的代价」(用例 6 反面教材)。

# 事件七:**User 提议「加锁就好了吧」 → mpac 0.2.8 + mpac-mcp 0.2.12**

> 0.2.11 部署后跑测试,User 看到"两个 Claude 同时启动 race window"问题,
> 直接说:**"那能不能加一个锁之类的?"**

这是几乎所有遇到 race condition 的工程师的第一反应。但"全文件锁"在分布式协议
有几个坑:

## "全锁"的 5 个问题

| # | 问题 | 具体场景 |
|---|---|---|
| 1 | **死锁** | Alice 锁 db.py + 想拿 api.py 的锁;Bob 锁 api.py + 想拿 db.py 的锁。两边互等 |
| 2 | **Lock granularity 选不对** | 文件锁 → 锁不住跨文件 dependency;symbol 锁 → announce 时 symbols 可能未知;project 锁 → 全员排队 |
| 3 | **Liveness / 锁释放** | Alice's Claude 拿锁后崩溃 → 锁谁来释放? |
| 4 | **跟现有 OPC 重复** | MPAC 在 commit 那层已有 `state_ref_before`(STALE_STATE_REF),问题是太晚 |
| 5 | **跟协议哲学冲突** | MPAC 故意 advisory,全锁等于把 MPAC 改成 mutex 服务 |

## 但 User 直觉的另一半对:**分级锁**

继续 0.2.11 的 git 类比 —— **scope_overlap 强保护 / dependency_breakage 弱提醒**:

| Conflict 类型 | 0.2.11 现状 | 0.2.12 提议 |
|---|---|---|
| **scope_overlap**(同文件)| Claude prompt 默认 yield(但 race window 失效)| **协议层硬拒** —— announce 时如果同文件已有 active intent,coordinator return `STALE_INTENT`,Claude 收到 reject → 不写文件 → 调 defer_intent |
| **dependency_breakage**(跨文件)| Claude prompt 默认 proceed + ⚠️ warning(race window 失效)| **announce 仍 accept,但 response 同步带 `conflicts` 字段**,Claude 看到非空后下一句话加 ⚠️ warning |

**核心区别 vs 全锁**:
- 不锁跨文件 dependency → 没死锁问题
- 锁释放 = intent withdraw / liveness timeout(已有机制)
- 跟 OPC 互补:OPC 在 commit 阻 race,新机制在 announce 阻 race
- 不违反 advisory 哲学:scope_overlap 本来就被认为是「必须 resolve」(同 git merge conflict)

# 自我观察

- **协议演进:从 advisory → 分级锁** 是个清晰升级。git 的 mental model 把哲学
  讲明白:**merge conflict 必须 resolve / semantic conflict 留给 CI**。MPAC 正好
  两套机制对应,**不需要发明新的协调哲学,直接借鉴**。

- **测试驱动协议设计**:今天每个协议改动都是测试触发的:
  - 用例 4 round 1 失败 → mpac-mcp 0.2.10 prompt MUST defer_intent
  - 用例 5 + user 问 "为什么不冲突" → mpac-mcp 0.2.11 split defaults
  - User 问 "加锁不就好了" → mpac 0.2.8 race lock + mpac-mcp 0.2.12 announce-response branches
  - Race 复现失败 + ⚠️ warning 没出来 → fake-browser observer → ab/ba bug 修复(225039f)
  
  **没有事先 design,完全是 user 测试 + reactive evolution**。

- **一晚上 5 个 PyPI 版本 + 5 次 Lightsail rebuild** —— 这是协议设计的高速迭代节奏。
  每个改动 commit + build + upload + sync_opensource + sync_deploy + ssh rebuild
  < 10 分钟。daily report 4-28 提到的 "LLM 时代的 baseline cadence" 再次复现。

- **fake-browser ws observer 是 prod debug 神器**:14 envelopes 一目了然,比让 user
  截 DevTools 快 10 倍。复用 4-26 daily report 同款 pattern。**可以把它作为
  reusable script 放到 `scripts/` 目录,以后每次 prod 怀疑 routing 问题第一时间用**。

- **架构限制要诚实**:Claude 子进程缺 reactive event 是 0.2.x 协议的 hard 限制。
  今天 0.2.12 把 "announce 时附 same-tick conflicts" 作为部分补救,但只覆盖
  后到的 announcer。先到的永远看不到。**Option C 是真正闭环,但是 0.3.x**。
  这种限制要明白讲给 user 听,不能假装能解。

- **同一 commit 揉太多改动是个坏味道**:`ff99707` 把 round 2 + round 3 + 又一轮
  fast-resolve fix 都揉在一起,git history 看起来不清晰。下次类似情况,**每个
  独立可验证的改动单独 cut commit**(即使没发版)。

- **PyPI 索引延迟** 这次 build api 时第一次失败(因为 PyPI Fastly 没追上),等
  90s 自动 OK。考虑给 `sync-deploy` script 加个 PyPI version check pre-flight,
  避免 Lightsail 端 build 失败重试。或者给 build 加 `--retry`。
- **测试者本机 mpac-mcp 升级时 relay 进程要 kill 重启** —— 4-28 已经踩过这个坑。
  今天我帮 user kill 了 2 次(0.2.10 升级时 / 0.2.12 升级时)。考虑给 bootstrap
  脚本加一个 `--kill-existing` 选项,或者 watchdog 自动重启 relay。
