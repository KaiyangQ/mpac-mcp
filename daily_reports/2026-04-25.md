## P1.3 — 浏览器 editor 走 MPAC 协调（最严重的卖点崩坏）

**Before**：[page.tsx](web-app/app/src/app/projects/[id]/page.tsx) 的 `handleEditorChange` 每个 keystroke debounce 800ms 后直接 `PUT /files/content`，没 intent、没 OCC、没 scope overlap 检查。Claude 在改一个文件时人类继续敲键盘，800ms 后人类的版本悄无声息盖了 Claude 的工作。MPAC "no silent overwrites" 的承诺崩了。

**After**：
- 后端 [`files.py:_assert_caller_holds_intent`](web-app/api/routes/files.py)：PUT/DELETE 强制要求调用方在 coordinator 里持有一个 scope 覆盖该 path 的非终态 intent；没有就 409 `INTENT_REQUIRED`。Greenfield 让步：项目从来没人连过 WS 时绕过（没人需要保护）。
- 前端：editor 第一下键入触发 `session.beginTask([path], "Editing X (browser)")`，intent_id 缓存在 `intentByPathRef`；切文件时 `yieldTask` 旧的；unmount 时全部 yield。create / delete 文件路径同样套这个 begin → 操作 → yield 模式。

## P2.1 + P2.2 — 项目级 lifecycle event channel

**Before**：删项目只清了 DB + dropped registry entry，活着的 WS 客户端继续握着 stale `ProjectSession` 引用直到下次握手才发现 token 没了。文件改动 / reset-to-seed 只 return 给 caller，其他浏览器要 F5 才看到新文件。两条都让 TC-4 / TC-6 的"3 秒内 peer 反应"成了空头支票。

**After**：
- 新桥能力 [`mpac_bridge.py:broadcast_project_event`](web-app/api/mpac_bridge.py) —— 同步 HTTP 路由用它把事件信封 fan-out 到所有 WS。事件 4 种 `kind`：`file_changed` / `file_deleted` / `reset_to_seed` / `project_deleted`。同步路由能调它的关键是：`SessionRegistry` 现在在第一次 async entry（如 WS HELLO）时捕获 running event loop，sync 路径 `asyncio.run_coroutine_threadsafe` 上来就能用。
- `lifecycle_delete_sync(project_id, name)`：单 coroutine 串行做"广播 → 关所有 WS → drop registry"，避免两步分开调度时的 close-先到-broadcast-空 race。
- `_ConnectedParticipant` 加了可选的 `close_ws` 回调，[main.py](web-app/api/main.py) 和 [ws_relay.py](web-app/api/routes/ws_relay.py) 注册 WS 时各自传一个 lambda 包 `ws.close(code, reason)`。
- `force_close_principals_sync` 给 `/leave` 用：只关 leaving user 的两条 principal（browser + agent）。
- 前端 [`useMpacSession`](web-app/app/src/lib/mpac-session.ts) 新接 `onProjectEvent` 回调，`PROJECT_EVENT` 信封触发它；page.tsx 里：
  - `file_changed` → 当前 open 的话 refetch 内容
  - `file_deleted` → 从树里移除，open 的切到下一个
  - `reset_to_seed` → 重 fetch /files + 清缓存
  - `project_deleted` → setLoadError + `router.replace('/projects')`

PUT/DELETE files.py 路由顺手在写完后调 `broadcast_project_event(..., exclude_principal=f"user:{user.id}")`，自己排掉自己（避免回环 refetch）。

## P1.1 — Agent token project-scoping 强制

**Before**：[`auth.py:get_user_or_agent`](web-app/api/auth.py) 收一个 token 就还原成 User，下游 `_assert_member` 只看 `user_id × project_id` 有没有非撤销的 token。如果 Alice 在 project A 和 B 都是成员，A 的 agent token 泄漏后能直接拿来访问 B（因为 Alice 在 B 也有 token）。docstring 自己都写了"handler is expected to cross-check"，结果谁都没 cross-check。

**After**：
- 引入 dataclass `AuthCtx(user, token_project_id)`：JWT 路径 `token_project_id=None`，agent token 路径填 token 行的 `project_id`。
- 新 helper `assert_token_scope(ctx, project_id)`：如果 `token_project_id` 不为 None 且不等于请求的 project_id 就 403。
- `files.py` 全 4 路由 + `agent.py` 4 路由（announce/withdraw/overlap/list_active_intents）都先 `assert_token_scope` 再业务逻辑。
- JWT 不受影响 —— 它本来就是 user-scoped 的，跨项目能力是设计意图。

## P1.2 — Query-string credentials 进 access log

**Before**：浏览器 WS `?token=<jwt>` + bootstrap 脚本 URL `?token=<agent_token>` 都进 nginx + uvicorn 的 access log。

**After**（最小风险路线，没动 cookie 鉴权那个大改造）：
- [`Dockerfile`](deploy/dockerfiles/api/Dockerfile) 给 uvicorn 加 `--no-access-log`。app 自己的 `mpac.*` logger 不受影响，结构化日志（HELLO accepted / GOODBYE 等）继续走 stderr → docker-compose logs。
- [`nginx.conf.template`](deploy/aws-lightsail/nginx.conf.template) 在 `/ws/` + `/api/projects/N/bootstrap.{sh,ps1}` 两个 location 加 `access_log off`。其他 `/api/*` 默认还是记，但都用 Authorization header，不带 query token。

后续大改造（cookie-based WS auth、bootstrap 用 nonce 换 token）记 TODO，没在这一波做 —— 那个要重写 WS 握手 + bootstrap 流程，风险太高。

## 没做的（明确推迟）

- **WS auth 切 cookie**：browsers 不能在 WS upgrade 设 Authorization header，cookie 是更干净的换法，但要解决 CSRF + cross-origin cookies + dev/prod 不同 SameSite 设置，体量太大单次 PR 装不下。`access_log off` 已经把"落盘"那一面堵住了，浏览器历史里 WS URL 通常不会留。
- **Bootstrap token 换 nonce 模式**：当前 user copy-paste `bash <(curl ...?token=xxx)`，应该改成"先换一次性 nonce、终端用 nonce 发 Authorization header 拉脚本"。要重写 mpac-mcp-relay 的 bootstrap 入口，等下一波。
- **uvicorn worker > 1 时的 lifecycle event 跨进程广播**：现在 `SessionRegistry` 是 process-local，prod 单 worker 没问题；要扩多 worker 得加 Redis pub/sub。Dockerfile 注释里已写"出 1 worker 时再外部化"，今天没动。

## v1 留下的 3 个真 bug

**state_machine.state 永远是 None**（[files.py](web-app/api/routes/files.py) + [agent.py](web-app/api/routes/agent.py)）：

**broadcast 排错了 principal**（[files.py PUT/DELETE](web-app/api/routes/files.py)）：

## P1.2 真正的收口（不再"日志脱敏当修复"）

reviewer 一针见血："access_log off 没真正收口" —— token 进 URL 这个事，对中间代理日志、`curl -v`、浏览器历史等还是裸的。

**Bootstrap 切到 Authorization header**（[`agent.py`](web-app/api/routes/agent.py)）：
- 新生成的 launch_command：
  ```bash
  bash <(curl -fsSL -H "Authorization: Bearer xxx" '.../bootstrap.sh')
  ```
  ```powershell
  iex (irm '.../bootstrap.ps1' -Headers @{Authorization='Bearer xxx'})
  ```
- 路由 `bootstrap.sh` / `bootstrap.ps1` 用新的 `_resolve_bootstrap_token(authorization, token_query)` helper，header 优先、`?token=` 兜底。已经粘到终端的旧命令仍能跑（不强切，让用户能 graceful 过渡），但 Connect Claude modal 不再生成 query 形式。

**浏览器 WS 切到 HttpOnly cookie**：
- `/api/login` 和 `/api/register` 现在多 set 一个 cookie：
  ```
  Set-Cookie: mpac_jwt=<jwt>; HttpOnly; SameSite=Lax; Secure (prod only); Max-Age=259200; Path=/
  ```
  （[`users.py:_set_jwt_cookie`](web-app/api/routes/users.py)）
- 浏览器 WS 升级时自动带 cookie，[`main.py:ws_session`](web-app/api/main.py) 改成 `cookie_jwt = ws.cookies.get("mpac_jwt"); raw_jwt = cookie_jwt or token`，cookie 优先、`?token=` 仍兜底。
- 前端 [`mpac-session.ts`](web-app/app/src/lib/mpac-session.ts) 把 `?token=` 从 WS URL 拿掉了。
- 前端 [`api.ts`](web-app/app/src/lib/api.ts) 加 `credentials: "include"` 让跨端口（dev: 3000↔8001）也能收发 cookie。后端 CORS 已有 `allow_credentials=True`。
- 新增 `POST /api/logout`：[`users.py:logout`](web-app/api/routes/users.py) 清 cookie。前端 [`auth-context.tsx:logout`](web-app/app/src/lib/auth-context.tsx) fire-and-forget 调它（不阻塞登出动画，失败也 OK 因为 origin 检查仍兜底）。

依旧没动 HTTP `/api/*` 的鉴权方式 —— 那条还是 Authorization header + localStorage JWT，本来就不在 URL 上，不在这一波的火力里。把 HTTP 也切成 cookie 是另一波（涉及把 JWT 完全从 localStorage 撤掉，XSS 收口的另一个独立故事）。

## 还推的（这次写明白）

- **WS cookie 的 SameSite=None+CSRF token**：当前 SameSite=Lax 在同域（prod）和 same-site（dev localhost 不同端口）都工作；如果未来前端搬到不同 eTLD+1（比如 frontend 上 Vercel、backend 在 duckdns），要么改回 query token、要么 SameSite=None+Secure+CSRF token。今天不动，因为现在没这个拓扑。
- **Bootstrap 的 nonce-exchange**：reviewer 还提了 "短时一次性 nonce 换 token"，比 header 形式更激进。当前 header 形式已经把 URL leak 堵了，nonce-exchange 主要是 defense-in-depth（即便 token 进了客户端 shell history 也 within 30s 失效）。要重写 mpac-mcp-relay 的 bootstrap 入口，单 PR 装不下，下一波。
- **HTTP `/api/*` 撤 localStorage JWT，全切 cookie**：能彻底防 XSS 偷 JWT。但要加 CSRF token、改所有 fetch 调用、改 logout。独立故事。
