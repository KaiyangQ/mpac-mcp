## 改动明细(今日 4 个 commit)

### `b2a09c6` mpac-mcp 0.2.0: relay auto-reconnect + PyPI publish

**问题**:昨晚的 relay 在 WebSocket 断开时直接 `return 0`,进程退出。实际生产里 FastAPI 重启 / 用户 WiFi 抖一下,relay 就挂,用户要手动重跑命令 —— 这不是"长驻 sidecar"该有的体验。

**修法**:把 WebSocket 连接逻辑套进一个 `while True` 循环,用 exponential backoff(2, 4, 8, 16, 32, 60 秒封顶)重连。只有两种情况才不重连:
- WS 握手返回 401 / 403 —— token 明确失效,重连也是白费(需要从浏览器重新 mint)
- `KeyboardInterrupt` —— 用户 Ctrl+C

其他任何 `ConnectionClosed / OSError / ConnectionRefusedError` 都进入 backoff 路径。连接成功后 `attempts=0`,下次断开从头算起。

顺便调了 `websockets.connect` 的几个 kwargs:
- `open_timeout=15` —— 握手期 15 秒超时,避免"假连上"
- `ping_interval=20, ping_timeout=20` —— ~40 秒内能探测出静默断开

跑本地 smoke test 时主动 `kill` 掉 API 然后再启,relay 能自己抓回来、Claude 重新出现在 `WHO'S WORKING` —— 验证通过。

### `706895e` web-app: agent-token reuses live token when relay is connected

**问题(昨晚踩过的坑)**:用户第一次点 "Connect Claude" → mint 新 token → 运行 relay → 一切正常。用户第二次点 "Connect Claude"(哪怕只是想再 copy 一次命令)→ 后端按原逻辑 revoke 旧 token + mint 新 token → 正在跑的老 relay 的 WebSocket 还活着(握手时已认证),但它传给 `claude -p` 子进程的 `MPAC_AGENT_TOKEN` env 变成失效的老 token → 所有 MCP tool HTTP 调用 401 → 一个迷之"chat 看起来没事但 Claude 什么都编不对"的状态。

**修法**:把 `POST /api/projects/{id}/agent-token` 的逻辑改成:

```
if relay_registry 里已经有这个 (user, project) 的活 relay:
    return 该 relay 正在用的 token(不 revoke,不 mint 新的)
else:
    revoke 所有旧 agent token + mint 新 token(原行为)
```

把 revoke-and-mint 拆到 `_rotate_agent_token` helper 里备用。

**UX 变化**:用户再次打开 modal 看到的是**和正在跑的命令一模一样的内容**。想强制换 token?停掉 relay 再开 modal。语义更可预期。

**副作用**:从安全角度看 token 不再被"强制轮换" —— 如果担心泄露,Settings 里该给个"Revoke now" 按钮(暂未加,记在 backlog)。
