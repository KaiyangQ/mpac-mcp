## Phase D：真实 MPAC coordinator，真实 presence + conflicts

### 架构决定（今天最重要的一个）

浏览器参与 MPAC session 有三条路：

| 方案 | 形态 | 问题 |
|---|---|---|
| A. 浏览器直连 fly.io hosted coordinator | 用昨天部署的 `mpac-demo.fly.dev` | 每个 (user, project) 的 MPAC bearer token 要注入 fly.io 的静态 `MPAC_TOKEN_TABLE` secret，每个新项目都要 `fly secrets set` + redeploy，扩展性 0 |
| B. FastAPI 后端 spawn 一个 `mpac-mcp-sidecar` 子进程 | 复用昨天的 sidecar | 需要写一个跨进程 verifier，多个移动部件 |
| **C. FastAPI 后端 in-process 直接用 `SessionCoordinator` 作 library** | 选中 | 零额外进程，auth 直接和我们的 SQLite `Token` 表对齐 |

选方案 C 的关键 discovery：`mpac_protocol.core.coordinator.SessionCoordinator` 是一个**纯 Python 状态机**，单入口 `process_message(envelope: dict) → list[dict]`，同步、无 I/O。不是 network 服务——是 library。研究 report 里写得很明白："SessionCoordinator 是协议核心引擎，纯 Python 业务逻辑……零 I/O 依赖；可在任何进程直接调用。"

所以 FastAPI 只需做一个薄桥：
1. 每个 project 拥有一个 `SessionCoordinator` 实例
2. 浏览器 WebSocket 握手时校验我们自己的 JWT（query param）
3. 查 DB 里的 MPAC bearer token，合成 HELLO 发给 coordinator
4. 路由 coordinator 输出回 WebSocket

这个决定从"demo 能不能做"一下子变成"很自然就能做"——后续 E phase 也是这个基础架构的直接延伸。

### 后端：`api/mpac_bridge.py`（新，~280 行）

核心数据结构：

```python
class ProjectSession:
    project_id: int
    mpac_session_id: str
    coordinator: SessionCoordinator
    connections: dict[str, _ConnectedParticipant]  # principal_id → conn

class SessionRegistry:
    # Per-process singleton, lazy-init per project on first connection
    async def get_or_create(project_id, mpac_session_id, verifier) -> ProjectSession
```

路由规则 **逐字 copy 自 `mpac_protocol.server.MPACServer.handler`** 以保持和参考实现行为一致：

| 出口 | 处理 |
|---|---|
| 入站 `INTENT_ANNOUNCE / INTENT_WITHDRAW / CONFLICT_ACK / OP_COMMIT / RESOLUTION` | 原 envelope 重广播给其他参与者 |
| 出站 `SESSION_INFO` / `PROTOCOL_ERROR` / `OP_REJECT` | 只给 sender |
| 出站 `CONFLICT_REPORT` | 只给两个 involved principal（`principal_a`、`principal_b`）|
| 其他 | 广播 |

关键细节：`SessionCoordinator` 在 `process_message` 中是**同步**的，FastAPI WebSocket handler 是**异步**的——直接在 async 里调 sync 没问题，因为 coordinator 是纯 Python 计算，不阻塞事件循环。

### Bridge 的一个小修（display_name 丢失问题）

第一轮测试时发现：Bob 通过 Python client 接入后，Alice 的面板里显示"user:2"而不是"Bob"。

根因：MPAC 的 HELLO 只给 sender 返回 `SESSION_INFO`——**其他参与者完全不会收到"某某加入了"的通知**（SPEC 把这留给应用层）。所以 Alice 第一次看到 Bob 是通过 Bob 的 `INTENT_ANNOUNCE`（sender 里只有 `principal_id`，没有 `display_name`）。

修复：在 [mpac_bridge.py](web-app/api/mpac_bridge.py) 的 `register_and_hello` 里加两段桥自合成的广播：
1. 新人 HELLO 成功 → 向其他连接广播一个合成的 `PARTICIPANT_UPDATE`（带 display_name / roles / principal_type）
2. 向**新人自己**重放所有已连接参与者的 `PARTICIPANT_UPDATE`（catchup），让 newcomer 立即拿到完整的 presence

前端 `useMpacSession` 已经支持 `PARTICIPANT_UPDATE`，零改动。

### Bridge 的另一个小修（agent 累积 offline）

每次 Claude chat 用的是新的 `principal_id`（`agent:claude:{project_id}:{uuid}`）——避免和前一轮的残留碰撞。但问题是：每次一轮完 Claude 的 PARTICIPANT_UPDATE(offline) 进来，前端**把他留在列表里显示"offline"**，导致多轮聊天后面板里躺着一堆"Claude AI offline"历史残骸。

修复：前端 `useMpacSession.markParticipantOffline` 逻辑改为：
- 如果是 agent → 从列表中**移除**
- 如果是 human → 保持在列表但标 offline（指示"刚才在这"）

## Phase E：Claude 作为 MPAC participant（peer，不是 copilot）

### 设计

关键设计决定：**agent 不是通过 WebSocket 接入，是通过 in-process loopback**。

为什么：agent 和 FastAPI 在同一个 Python 进程里，没必要走 WS。我们让 agent 的"连接"是一个 async callable `send(envelope: dict) → None`——对 bridge 来说和真 WS 连接完全同形（duck-typed）。

Ephemeral agent token：每次 chat 生成一个随机 token（`secrets.token_urlsafe(24)`），加到 `mpac_bridge.agent_tokens` 这个 per-project in-memory allowlist 里，agent run 结束就删。coordinator 的 verifier 先查 DB 里的 `Token` 表，然后查这个 allowlist，都不中 → 拒绝。

这样：
- 不需要 DB schema 变更（`Token` 表没有 `is_agent` column）
- Agent 的凭证生命周期完全由代码控制，泄漏风险 0
- 协议层还是严格走 Authenticated profile（credential 被真实 verify）

### `api/agent/claude_agent.py`（新，~400 行）

一次 chat turn 的完整流程：

```
1. JOIN session（合成 HELLO） — 立即，Claude 出现在面板为 idle
2. Stream Anthropic API — 实时解析 tool_use 的 input JSON
3. 一看到 files 数组闭合 → announce_intent（这是别人开始看到 Claude 在干啥的瞬间）
4. 继续 stream summary —— intent 被"真的"占用中
5. Stream 结束 → withdraw_intent
6. GOODBYE
7. Return summary 作为 chat 回复
```

Tool schema：单次强制 tool_use `announce_work`，字段顺序 `objective → files → summary`（见下面"流式重构"段落说明为什么是这个顺序）。

### 流式重构：真实的工作窗口

改用 `client.messages.stream()` 方案：

```python
async with client.messages.stream(...) as stream:
    accumulated_input = ""
    announced = False
    async for event in stream:
        if event.type == "content_block_delta" and event.delta.type == "input_json_delta":
            accumulated_input += event.delta.partial_json
            if not announced:
                objective = _extract_objective(accumulated_input)
                files = _extract_complete_array(accumulated_input, _FILES_KEY)
                if objective and files:
                    # 文件列表解析完了，立即 announce
                    announce = participant.announce_intent(...)
                    await process_envelope(session, announce, principal_id)
                    announced = True
    
    final = await stream.get_final_message()

# Stream 结束 → 立即 withdraw
if announced:
    await process_envelope(session, withdraw_intent_msg, principal_id)
```

关键点：
1. **tool schema 的字段顺序改为 `objective → files → summary`**。Claude 按 schema 顺序生成 JSON，所以 `files` 会在 `summary` 之前就完整出现——announce 能早早触发。
2. **增量 JSON 解析**：写 `_extract_complete_array` 走字符级解析（跟踪 string 内外、escape 状态），判断数组是否真的闭合。
3. **intent hold 时长 = "files 解析完" 到 "summary 流完" 的真实时间**——完全由 Claude 决定。长 summary 窗口就长，短 summary 就短。

    level=os.environ.get("MPAC_LOG_LEVEL", "INFO"),

## 今天踩到的坑 / 学到的东西

1. **FastAPI 的 header 要用 `Header()` 标注**——不然 `authorization: str = ""` 被当 query param，request 里读不到。Phase A 只 smoke import 过，没跑真实 HTTP，所以 bug 藏了一整天才暴露。教训：每一层 stub 接口都应该至少跑一次端到端 curl。

2. **macOS 系统 Python 3.9 不支持 `X | None` 类型语法**——`from __future__ import annotations` 能糊住大部分场景，但 FastAPI / pydantic 会 `eval()` forward refs 时炸。解决：`/opt/homebrew/bin/python3.12` 重建 venv。

3. **Claude Code OAuth token 走不通公共 `/v1/messages`**——即使 scope 包含 `user:inference` 也会 401。需要 Claude Code 特定的 `anthropic-beta` header，而且 TOS 上是灰色地带。教训：用自己生成的 `sk-ant-api03-` key 才是正路。

4. **浏览器 setInterval 被背景标签节流**——我第一次写的"每 500ms 采样 panel state"代码在未 focus 的标签里变成每 1-2s 一次，错过了 Claude 几秒钟的 work window。教训：demo 录制 / 自动化测试里需要**观察事件本身**（从 backend log 里抓 `announced intent early` timestamp），不要靠前端 polling。

5. **MPAC 不自动广播"谁加入了"**——HELLO 只给 sender 返回 `SESSION_INFO`，其他人第一次知道新参与者是通过对方的 `INTENT_ANNOUNCE`（envelope 里只有 `principal_id`，没 `display_name`）。应用层需要自己合成 presence 广播，或者从 SESSION_INFO 里读 `participants` 数组。

6. **`CONFLICT_REPORT` 只发给两个 involved principals**——`MPACServer` 和我们的 bridge 都这样做（for 效率和隐私）。所以第三方观察者看不到冲突。这个语义在 demo 里需要注意：要展示冲突，Alice 或 Bob 必须是冲突的一方。

7. **SSE stream 在 Anthropic SDK 里的事件类型**：`content_block_delta` + `delta.type == "input_json_delta"` 才是 tool_use 的字段流，不是 `"text_delta"`。第一版代码用错事件类型导致没触发 early announce，debug 10 分钟。

8. **JSON schema property 顺序决定 tool_use 生成顺序**——Claude 生成 tool_use block 时严格按 schema 的 properties 顺序写字段。把 `files` 放在 `summary` 之前是今天能做出 early announce 的前提。

9. **找 API key 的好地方**：项目根的 `local_config.json`、macOS keychain（Claude Code OAuth）、VS Code workspaceStorage 的历史缓存、`~/.zshrc`。今天最意外的是 VS Code cache——它会保留所有你在 VS Code 里编辑过的 `.env` / `.env.example` 文件的备份。

10. **凭一行 log 看出 agent 的生命周期**：今天最后那两条日志 `announced intent early` → `GOODBYE`，中间的时间差**就是**"Claude 在这个 session 里可见地工作的时间"。把这个量变成可观测数据（而不是黑盒）对产品 debug 和 demo 都很有价值。
