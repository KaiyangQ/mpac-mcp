# 事件四:发现 #2 — scenario 7 ⚠️ prompt rule 听话率低

scenario 7 真 claude e2e:

```
05:53:48 user=1 alice announced db.py + symbols=[notes_app.db.save] same_tick_conflicts=0
05:53:58 user=2 bob   announced api.py + symbols=[api.save_with_log]  ← same_tick_conflicts=1 ✓
```

`same_tick_conflicts=1` 证明 server 正确触发了 dep_breakage,bob 的 announce
response 同步带 `conflicts: [{...their_impact_on_us...}]`(昨天 ab/ba fix
那条 code path)。

但 bob 的 chat reply **完全没出现 ⚠️**:

> done。已在 `notes_app/api.py` 第 29-32 行添加了 `save_with_log(note)` 函数:
> ```python
> def save_with_log(note: Note) -> None:
>     db.save(note)
>     print(f"[LOG] saved note id={note.id} title={note.title!r}")
> ```

直接吞了 conflicts 字段的提示价值 —— 这是 0.2.11 prompt 第二个 branch
("dep_breakage default PROCEED with prominent warning,reply 第一段 START
with: ⚠️ {other} is changing `{symbol}` ...")**完全没被听话执行**。

对比同时段的 race-lock-rejected 路径(下面 6 条 server log,全是 STALE_INTENT
触发后 bob 的 claude 调 defer + chat 解释 + 提供 override 选项):

```
05:06:15 reason="另一位参与者正在编辑 db.py，我先让路"
05:09:58 reason="scope_overlap: agent-1 正在编辑同一文件，让路等待"
05:12:59 reason="另一位参与者在 check_overlap 和 announce 之间抢先锁定了 db.py"
05:16:26 reason="scope_overlap: Alice's Claude 正在编辑同一文件，让路等待"
05:19:45 reason="scope_overlap: Alice's Claude 正在编辑同一文件，让路等待"
05:57:02 reason="scope_overlap: Alice's Claude is editing notes_app/db.py"
```

**rejected 路径听话率 6/6 = 100%**;**warning 路径听话率 0/1 = 0%**(样本
小,但跟 6/6 对比足够 suspicious)。

## 怀疑原因

rejected 路径 mpac-mcp 的 `announce_intent` 工具返回 `{"rejected": true,
"error_code": "STALE_INTENT", ...}` —— 是个**显眼的 sentinel** 形结构,
prompt 里有专门的 rejected branch 接着,Claude 一眼就能识别。

warning 路径返回 `{"intent_id": "...", "accepted": true, "conflicts": [...]}`
—— `conflicts` 只是 success response 里多了个字段,prompt 也是个 inline
分支("if conflicts non-empty → ⚠️"),没显眼 sentinel。Claude 容易把
`conflicts` 当 "metadata" 而不是 "must-surface"。

## 0.2.13 prompt 改进方向(backlog)

两条:
1. mpac-mcp `announce_intent` 工具如果 `conflicts` 非空,把 return 包成
   `{"accepted": true, "conflicts": [...], "user_action_required": "PREFIX_REPLY_WITH_WARNING"}`
   —— 加显眼 sentinel
2. tool docstring 里加 MUST 级条款:"⚠️ if conflicts non-empty"

doc 里也加了"2026-04-30 e2e 实测确认这是真问题" caveat。

# 事件五:发现 #3 — override "硬上" 第二轮 claude 不重试

3 phase override flow(alice API 60s hold + bob 真 claude 3 轮 chat):

| Phase | Bob 输入 | 期望 | 实测 |
|---|---|---|---|
| 1 | 写 2 函数 | check_overlap → 看到 alice → defer + chat 解释 + 提供 override | ✅ 完美,reply 包含「硬上」/「proceed anyway」选项 |
| 2 | "硬上"(alice 还在持有)| retry announce → STALE_INTENT 又拒 → chat 解释二次拒+让用户等 | ✅ 完美,reply:"服务器端 race lock 硬拒,我没法绕过" |
| 3 | "硬上"(alice 已 withdraw 1s)| retry announce → 这次会成功 → 写文件 | ❌ **claude 根本没调 announce**,直接复用 phase 2 的 cached STALE_INTENT 答案 |

phase 3 的 chat reply 关键证据:

> 理解你想强行推进，但这是服务器端的硬锁（`STALE_INTENT`），不是我这边能绕过的——
> 协调器直接拒绝了 announce 请求。无论我重试多少次，**只要 Alice's Claude 的 intent
> `intent-agent-1-f8386ab208` 还活着**，服务器都会拒绝。

里面**直接引用了 phase 2 的 intent_id `intent-agent-1-f8386ab208`** —— 这个
intent 在 phase 3 fire 之前 1 秒已经 withdraw 了。Claude 完全没去 retry
确认状态,基于过期的 turn-context 做决定。

server log 印证:phase 3 时段(05:57:47 - 05:57:53)**完全没有新 announce
HTTP 进来**,不是 server 拒了,是 claude 根本没问。

## 这就是昨天 daily 提的 Option C 限制的具体表现

mpac-mcp 的 `claude -p` 子进程在 turn 之间没有 reactive event subscription,
也不会在 user 输入 turn 启动时 refresh world state。Claude 完全靠**上轮 tool
call 留下的 context** 决定本轮要不要再调工具。

phase 2 留下的 context:"alice's intent f838... 还在,announce 被 STALE_INTENT
拒"。phase 3 user 说 "硬上" → claude 想"上轮就是这个回答,这次还是这个,直接
复用" → 不调 announce_intent。

## 修法

短期 (0.2.13 prompt level):明确告诉 claude
> When user says "硬上" / "proceed anyway" after a STALE_INTENT, ALWAYS
> re-call `announce_intent` first — the holding intent may have been
> withdrawn since you last checked. Never assume prior tool results
> still hold.

长期 (0.3.x Option C):relay 推 INTENT_WITHDRAW 到 claude -p 的下一轮 turn
context,让 claude 真"知道" alice 已经走了。
